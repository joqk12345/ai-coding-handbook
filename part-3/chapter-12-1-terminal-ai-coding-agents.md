# 12.1: 构建终端原生 AI 编码智能体：脚手架、运行框架（Harness）、上下文工程与经验教训

大语言模型（LLMs）的迅猛进展，催生了软件开发中的一种新范式：以 AI 为驱动、能够作为自主智能体运行的编码助手 [92, 102, 96]。与传统只提供行内补全建议的工具不同，这类 agentic coding assistants 能够围绕复杂任务进行推理、执行多步骤计划，并通过工具调用与开发环境交互。关于代码智能的综述研究已经系统记录了这一领域的爆发式增长 [44, 43]，而 Agentic Software Engineering 路线图 [1] 则进一步把人机协作的方法论明确化。SWE-Agent [92]、OpenHands [81] 与 HyperAgent [66] 等系统，已经在标准化基准上展示了自主智能体的潜力。商业层面的变化同样剧烈：GitHub Copilot 的开发者用户已超过 1500 万 [29]，AI 原生编辑器收入快速增长 [5]，各大实验室也相继推出自主编码智能体 [18, 62]。这些信号表明，Agentic Coding 已经从研究原型走向工业部署。

## 终端原生智能体的崛起

在过去几年里，AI 编码助手主要以 IDE 插件的形式存在，扮演需要人类持续盯防的响应式 Copilot。最近，一场重要转向已经出现：业界开始从复杂的 IDE 集成，回归到更简单、更通用的命令行界面。Claude Code [3] 引领了这一变化，它证明了终端原生智能体在真实软件工程任务中的表现，可以与甚至超过 IDE 内置工具。终端本身就是软件开发的操作核心，它天然承载了版本控制、构建系统、远程 SSH 会话以及无头服务器环境。Aider [27]、CodeAct [80] 与 Open Interpreter [60] 等更早的系统，也已经证明了基于终端的 AI 结对编程与可执行代码操作是可行的。如今，几乎每家主要 AI 实验室都提供了 CLI 智能体 [3, 30, 62]，开源阵营中也出现了 Goose [10]、OpenCode [19] 与 Crush [13] 等替代方案。

但要把这种潜力真正转化为可用产品并不简单。Terminal-Bench [54] 与 LongCLI-Bench [25] 等基准显示，即便是前沿模型，在持续终端操作场景下仍然表现吃力。这说明，终端原生 Agent 不能只依赖更强的模型，还需要面向场景的工程化设计。

## 长时运行终端 Agent 的三大挑战

这些基准结果共同指出，任何长时间运行的终端智能体，都必须解决三个基础工程问题。

第一，如何在会话长度持续超过模型 token 上限的情况下，管理有限的上下文窗口。终端任务往往是多轮、跨文件、跨命令的，单次上下文并不足以覆盖全部历史。

第二，如何在智能体可以执行任意 shell 命令时，阻止破坏性操作。终端能力越强，误删文件、错误提交、污染环境等风险也越高。

第三，如何在不挤爆提示词预算的前提下扩展系统能力。工具越来越多、指令越来越长，如果所有能力都直接塞进 prompt，系统很快会变得臃肿且不稳定。

围绕这些问题，我们把终端智能体的体系结构分成两个阶段：其一是 scaffolding，即在第一轮提示之前完成智能体装配，包括系统提示词、工具 schema 与子智能体注册；其二是 harness，即运行时负责工具调度、上下文管理与安全约束的执行框架 [99]（见第 2.2 节）。然而，终端原生 Agent 工具的设计空间至今仍缺乏系统研究。大多数生产系统是闭源的，关键架构决策没有公开；已有开源框架要么主要面向 benchmark，而不是交互式日常使用，要么缺少成体系的技术报告 [53, 36]。因此，三个核心开放问题构成了本文的出发点：多模型架构应如何在成本、时延与能力之间做平衡？哪些安全机制既能阻止破坏性操作，又不会显著拖慢开发效率？系统又该如何在有限上下文中支撑长程对话？

## OpenDev 的定位与本文贡献

本文提出 OpenDev，一个面向软件工程的开放式 AI 命令行智能体。据我们所知，这是第一份围绕开源、终端原生、可交互编码智能体撰写的系统性技术报告。现有系统大致分为两类：一类是 SWE-Agent [92] 这类偏 benchmark 的框架，虽然有论文，但主要为自动化评测而设计，不强调日常交互使用；另一类是更接近生产的系统，例如 OpenHands [81]，虽具备较完整文档与产品成熟度，却依托浏览器界面而非终端。与此同时，Aider [27]、Goose [10]、OpenCode [19]、Crush [13] 与 Gemini CLI [30] 等 CLI 原生 Agent，又普遍缺少公开的技术报告来解释其设计权衡；Claude Code [3] 则虽然是 CLI 原生代表，但既不开源，也没有完整技术报告。

因此，本文的目标并不是提出一个全新的算法突破，而是公开一个可用于生产环境的 Agentic Coding 系统在工程实现上的关键决策、权衡与经验总结，用来弥合闭源工业实践与开放学术讨论之间的落差。

## OpenDev 作为复合 AI 系统

OpenDev 的核心设计原则之一，是把系统视为一个复合 AI 系统（compound AI system）[100]：它不是单一、整体式的大模型，而是由多个 agent 与工作流组成的结构化组合体，并且每个组成部分都可以独立绑定到用户配置的 LLM（见第 2.2.5 节）。这一框架与 Zaharia 等人的观点一致，即当前最先进的 AI 效果越来越依赖多模型、检索器与工具的组合，而不是一次单独的模型调用。

OpenDev 将这一原则工程化地落地。它的解耦架构使系统在设计上天然保持模型无关性，而类似 learned model routing [59] 的方法，也可以被施加在工作流层而非单个提示层。对系统而言，更换模型提供商或优化成本只需要调整配置，而不需要改动代码。也正因为如此，OpenDev 的能力不是在部署时一次性冻结的，而是能够随着更优模型的出现持续升级。
