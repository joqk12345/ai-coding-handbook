# 12.6: OpenDev 的五个核心设计张力与可迁移经验

前几节已经分别介绍了 OpenDev 的架构、上下文工程、工具系统与持久化层。这里不再按组件逐个说明，而是退后一步，总结塑造整个系统的五个跨组件设计张力。它们不是某个模块内部的局部问题，而是任何长时间运行的 agent 系统都绕不开的共同约束。

这五个张力分别是：

- 上下文压力作为核心约束
- 长时程行为引导
- 通过架构约束实现安全
- 面向“近似正确输出”的工具设计
- 懒加载与有界增长

每一节都会把多个组件串起来看，并提炼成可迁移的工程经验。

## 12.6.1 上下文压力才是中心设计约束

与 CPU 或内存不同，上下文预算既会被系统自身消耗，也会被 agent 自己的行为持续吞噬。系统提示词、工具 schema、安全前导文本会占用预算；文件内容、命令输出、搜索结果和对话历史同样会占用预算。更重要的是，这些部分竞争的是同一个有限窗口。

在 OpenDev 的实际运行中，工具输出通常会占掉 70% 到 80% 的上下文，远高于 system prompt 与 agent 自己的推理内容。这使得“上下文利用率”成为决定 agent 会话寿命的最关键指标。 richer tool outputs 确实能提升单轮准确率，但它们也会更快耗尽整个会话的可用寿命。

### 经验 1：把 context 当预算，不要当缓冲区

上下文压缩不应是一个“到阈值再全部压掉”的二元动作。更有效的做法是分阶段管理：持续监控利用率，在工具输出开始变旧时就先做 pruning，再做 masking，把真正的 LLM compaction 留给确实快溢出的情况。OpenDev 在 2.3.6 节里的 fast pruning 就是典型例子：它逆序扫描旧工具结果，把已经超出工作视野的输出替换为 `[pruned]`，经常足以避免更昂贵的 LLM 全量压缩。

直接教训是：不要只设计一个“紧急压缩阈值”，而要设计渐进式 reduction 阶段。

### 经验 2：把大输出卸载到文件系统

当某个工具输出超过阈值时，最划算的处理方式不是把它继续留在上下文里，而是写到 scratch 文件中，只返回一个短预览和文件路径引用。这样，agent 仍然能看到足够多的信息来判断“这个输出值不值得继续看”，而只有在真正需要时才通过一次 `read_file` 把完整内容重新取回。

这本质上是把一个“持续支付的上下文问题”转成了“一次性的检索问题”。后者显然更便宜，因为检索只在按需读取时付费，而上下文占用会在后续每一轮 LLM 调用中不断重复支付。

### Prompt 结构要为缓存服务

对支持 prompt caching 的 provider，把 system prompt 拆成稳定前缀和动态后缀，并只缓存稳定前缀，可以显著降低多轮会话的输入成本。原因很简单：system prompt 每一轮都会被重新发送，而其中最贵、最稳定的部分往往又是不会变的。

### Thinking 上下文要使用双记忆

如果 thinking model 直接吃完整对话历史，很快就会失控。更好的方式，是把长程上下文压缩成一段摘要，把短程细节保留为最近几轮原文，再把两者与当前问题一起拼进去。这样 thinking 预算始终是有界的，同时不会丢掉整体目标与最新操作细节。

这里有个细微但重要的点：不要反复“总结上一次的总结”。那样会产生累积漂移。应该定期从完整历史重新生成新的长程摘要，而不是继续压缩旧摘要。

### 经验 3：用 API 返回的 token 计数校准，而不是迷信本地估算

本地 token 估算很容易系统性低估真实用量，因为 provider 可能会在你看不见的地方插入安全前导、工具 schema 序列化或其它内部格式化内容。OpenDev 的实践表明，这种偏差足以让 compaction 触发得太晚，最终造成上下文溢出。

因此，上一轮 API 返回的 `prompt_tokens` 应该被当作地面真值；本地估算只适合作为没有历史 API 结果时的后备。

## 12.6.2 如何在长时程里持续引导行为

system prompt 的影响会随着对话增长而衰减。许多在前几轮严格遵守的规则，在 30 次甚至更多工具调用后会被稳定违反，因为这些规则已经被压到注意力窗口的远处，并埋没在大量工具结果之下。行为引导在本质上是一个信噪比工程问题：怎样在不淹没上下文的前提下，让模型持续遵守关键约束。

### 经验 1：提醒要在决策点注入，而不是在开头堆满

简短、单用途、接近当前决策点的 reminder，比一大段模型早已半遗忘的 system prompt 更有效。更关键的是，提醒应当作为 `role: user` 注入，而不是 `role: system`。在 OpenDev 的实验中，最近的 user-role 提醒显著比 system-role 提醒更能改变模型下一步行为，因为模型会天然赋予最新用户消息更高权重。

但提醒也不能无限注入。频率必须按类型加上上限，否则 reminder 自己会退化为背景噪音。

### 经验 2：把 thinking 与 action 分开

只要工具是可见的，LLM 就会更倾向于尽快行动，而不是先深思。给模型增加一个没有工具的 thinking phase，通常比在同一轮里说“请仔细思考后再行动”更有效。真正改变模型行为的，不是那句提示词，而是该轮 API 调用里根本没有工具 schema 这件事。

### 检索工具选择需要显式决策树

如果不告诉 agent 如何选工具，它几乎总会默认走 grep / text search。这会浪费大量迭代，并把大量假阳性结果灌进上下文。更有效的方法，是在 prompt 中显式编码检索决策树：

- 目标是函数名、类名：用 `find_symbol`
- 目标是字符串、报错文本：用 `text_search`
- 目标是结构模式：用 `ast_search`
- 目标是文件路径约定：用 glob / `list_files`

注意，这类规则必须是可观察、可执行的，而不是抽象地说“请使用最合适的工具”。

### Provider 条件段落值得单独建模

不同 provider 的能力边界非常真实，例如是否支持 extended thinking、function calling 细节、上下文长度等。把这些差异做成按 provider 条件加载的 prompt section，比写一堆“模糊的 provider 无关指令”更高效，也更省 prompt 预算。

### 经验 3：把 agent 的构造期和运行期彻底分开

长时运行 agent 的一个基础工程原则，是把 scaffolding 和 harness 分离。scaffolding 只负责在第一条 prompt 之前完成构建：生成 system prompt、注册工具、装配 subagent。harness 只负责运行时编排：工具调度、上下文管理、安全执行与状态持久化。

这种分离的直接收益是：构造期和运行期可以独立演化。新增工具时，你只需要修改注册表；调整 compaction 策略时，你只需要修改 harness。两者不会缠在一起。

## 12.6.3 通过架构约束而不是运行时判断来实现安全

如果把安全主要寄托在运行时权限判断上，模型依然能“看见”危险工具，依然可以围绕它推理、试探与寻找边界漏洞。更稳健的办法，是直接让某些行为在结构上不可能发生。如果写工具根本不出现在 agent schema 里，模型就不可能尝试调用它，因为它甚至不知道这种能力存在。

### 经验 1：不要只拦截危险工具，要让它对模型不可见

schema gating 比运行时 permission check 更鲁棒。一个不在 schema 里的工具：

- 不能被模型尝试调用
- 不能被模型“争取例外”
- 不能被模型拿来试探绕过路径

OpenDev 的安全系统之所以有效，不是因为某一层特别强，而是因为多层机制相互独立：schema gating 阻止越权尝试，approval 系统做人类审查，file freshness validation 防止覆盖并发修改，shadow git snapshots 确保所有文件系统变化可逆。某一层有 bug，不会直接削弱其它层。

### Approval 必须跨会话持久化

如果用户勾选了“总是允许某类操作”，但这个决定不会持久化到磁盘，那么用户每次重启都要重新审批同样的命令，最后只会导致 approval fatigue，进而逼着用户一键放开所有审批，安全系统被自己拖垮。

### Lifecycle Hooks 的接口必须尽早设计

外部 hook 是把策略、审计、CI 集成与安全增强开放给团队的关键手段。但 hook 如果没有从一开始就设计清楚：

- 阻塞与非阻塞语义
- 是否允许修改输入
- 全局与项目级配置如何合并

那它就只能成为一个“看起来有扩展性、实际很难用”的装饰接口。

### 模态优先级必须高于 agent interrupt

当用户在审批框或问答弹窗仍然激活时触发 interrupt，系统应优先取消弹窗，而不是直接打断 agent 线程。否则会留下悬空 spinner、未 resolve 的 future、失效的 UI 状态，最后还得手工清理。

## 12.6.4 为“近似正确输出”设计系统

LLM 生成的内容大多数时候不是完全正确，而是“接近正确”。编辑目标会与真实文件差一点点：尾部空格不同、缩进有偏差、转义不一致。错误恢复提示可能建议使用当前 agent 根本没有的工具。查询可能被路由到一个次优工具。若系统要求模型每次都绝对精确，它会把大量时间浪费在无意义的错误恢复循环里。

### 经验 1：工具本身要吸收 LLM 的不精确

`edit_file` 就是最典型的例子。严格 exact-match 的编辑工具，在真实环境里会制造大量失败，不是因为 agent 意图错了，而是因为它对目标文本的复现略有偏差。把这类误差建模为一串逐步放宽的 matcher，并让 matcher 返回“原文件中实际命中的子串”而不是查询文本本身，就可以把大量 near-miss 转成成功编辑，同时保留文件原始格式。

更普遍的原则是：如果 agent 的意图足够明确，而问题只在字面输出不精确，那么应该由工具桥接这道缝，而不是直接拒绝。

### 经验 2：恢复提示必须适配 agent 当前拥有的工具

当系统建议 agent 如何恢复时，不能给出一条“理论上正确、实际上当前 agent 做不到”的路径。若当前 agent 能委派 subagent，就提示它这么做；若它本身已经是 subagent，则应提示分块读取、offset/limit 之类的增量处理方案。

同样的原则也适用于错误消息本身。像“请重读文件，再基于当前内容重试编辑”这种指导，远比“请重试”有用，因为它把错误转译成了一条具体动作。

### 自动提升 server-like 命令到后台执行

开发服务器、watcher 和很多测试进程本来就不是设计成前台短命任务的。若一律在前台跑，它们迟早会撞上 timeout。通过正则识别 server-like 命令并自动转入后台，再配合输出捕获，是一种典型的“系统吸收环境不精确”的做法。

### 缺失依赖要自动安装，而不是用模糊错误把锅甩给用户

依赖外部 runtime 的工具，第一次运行时若发现依赖缺失，最佳做法通常是：

1. 检查依赖
2. 自动安装
3. 重新执行

而不是直接抛出一个 opaque error。这样能消灭一大类本不该由用户与 agent 共同承担的环境初始化失败。

## 12.6.5 懒加载与有界增长

一切 eager loading 在规模上来之后都会出问题。OpenDev 早期把所有 MCP tool schema 在启动时直接塞进 prompt，结果在第一条用户消息出现前就消耗了约 40% 的上下文预算。skills 也是类似问题：若启动时把完整技能内容全部加载，大部分会话都在为“永远不会用到的知识”付上下文成本。

### Lazy Discovery 把启动成本压到最低

MCP 工具与 skills 的共同解法是两阶段：

- 启动时只加载轻量元数据索引
- 真正使用时再加载完整内容

对 MCP 工具来说，这样可以把启动上下文成本从约 40% 压到 5% 以下。对 skills 来说，agent 启动时只知道“有哪些 skill、它们适用于什么触发条件”，但不会提前背下多页模板正文。

### 外部元数据缓存要支持 stale-while-revalidate

模型能力、价格和上下文上限这类外部元数据，也适合做 stale-while-revalidate 缓存：

- 缓存新鲜：直接使用
- 缓存过期：先用旧缓存，再后台刷新
- 刷新失败：继续使用旧缓存

这样系统就不会因为“暂时连不上 catalog API”而整体无法启动。对于 agent 系统来说，离线可启动性远比缓存内容是不是 100% 最新更重要。

### 经验 1：所有会随会话长度增长的资源都必须有上限

一切不设上限的资源，最终都会在长会话中出问题。典型例子包括：

- undo history 无限制累积
- 并发 tool calls 无限制膨胀
- behavioral nudges 无限刷屏
- agent iteration 无界循环

对应的原则很简单：凡是会随会话增长而线性或超线性增加的资源，都必须有 cap。

在 OpenDev 里，这表现为：

- iteration limit
- 固定上限的 undo history
- 并发工具调用 worker 上限
- 每类 reminder 的 nudge budget
- 只读调用可并行、写入调用串行

### 经验 2：阈值不要试图先验推导，优先从失败案例中调出来

context pressure 阈值、pruning 保护窗口、reminder 频率、iteration limit 这类数字，很难从第一性原理直接推导出来。它们依赖于模型行为、典型用户工作流与系统开销的交互。最有效的方法通常是：

- 从保守值开始
- 观察真实失败模式
- 持续回调阈值

OpenDev 中像 70% 的 compaction 触发线、3 次 nudge 上限，都是从实际失败分析里长出来的，而不是纸面推算出来的。

### 自愈索引要成为常态设计

凡是为了性能而引入的派生索引文件，都不应该成为新的单点故障。正确做法是：索引缺失或损坏时，能从底层原始数据自动重建。这样索引只是优化，而不是系统可用性的前提。

### 确定性操作不要硬塞进 agent loop

不是所有输入都值得走 LLM。像 session 管理、模式切换、模型选择、server 配置这些操作，本质上是确定性的，应该直接在输入边界处理，而不是浪费 token、引入 nondeterminism，再额外走一遍 approval 与 undo 逻辑。

## 小结

OpenDev 总结出的这五个设计张力，并不只属于它自己。长时程 agent 系统普遍会遇到同样的问题：上下文会被系统与工具共同吞噬，提示词影响会随时间衰减，安全不能只靠“最后一关拦截”，模型输出天然近似而非精确，而任何不设边界的增长最终都会在长会话中反噬系统。

因此，这一章真正想传递的，不是某个具体实现细节，而是五条更底层的工程直觉：

- 把上下文当预算管理
- 在决策点重新注入行为约束
- 让危险能力对模型不可见
- 让工具吸收模型的不精确
- 给所有会增长的资源设置边界

这些原则同样适用于更广泛的代码智能、上下文工程与自主软件工程系统。
