# 12.3: OpenDev 的上下文工程层：提示词、记忆、压缩与检索组装

基于 LLM 的智能体，并不是简单地把用户消息发给模型，再把模型回复返回出来。每一次回复的质量，首先取决于模型在当前上下文窗口中究竟能看到什么：它收到了哪些指令，保留了多少对话历史，从过去交互中学到了什么，以及在每次调用之前，系统如何把相关的外部信息组装进上下文 [53, 36, 98]。我们把负责管理这一整套上下文窗口的机制，统称为“上下文工程层”（Context Engineering Layer）[53, 36, 98]。

从系统流程上看，用户请求先经过 `QueryProcessor`，再交给 `ContextPicker` 组装上下文；组装后的内容进入 `ContextCompactor` 做 token 预算约束，然后才进入 ReAct 推理循环。每一轮执行完成后，结果会回流到 `SessionManager` 进行持久化，而其中那些具有“可学习价值”的工具结果，则会被自适应记忆子系统吸收与加工。也就是说，上下文工程层并不是单一模块，而是一个贯穿整个会话生命周期的协作网络。

## 12.3.1 动态系统提示词构建

在基于 LLM 的智能体中，system prompt 是行为控制的主要工具。智能体的身份、能力边界、安全约束、任务约定，几乎所有无法通过代码硬编码的行为属性，最终都要靠自然语言写进 system prompt。因此，启动时如何构造 prompt，并不是一个次要配置问题，而是整个系统的核心初始化问题。

一种常见但低效的做法，是直接加载一份“大一统”的单体 prompt，把所有可能用到的规则全部塞进去。这有两个叠加成本。第一，许多与当前会话无关的部分，例如非 Git 仓库目录里的 Git 工作流规则、未使用 subagent 时的调度说明、关闭任务跟踪功能时的 todo 规则，会白白占用上下文预算。第二，这些无关规则会稀释真正重要的指令，让模型行为变得更噪声化。正确的方向，不是手工删 prompt，而是让 prompt 的加载过程从一开始就是“上下文敏感”的。

### 按优先级排序的条件式拼接

OpenDev 在运行时组装每个 agent 的 system prompt。所有行为指令都被拆成独立 section，每个 section 以单独 markdown 文件存在，并注册两个元数据字段：

- 条件谓词：基于运行时上下文字典判断是否加载；`None` 表示总是加载。
- 优先级整数：决定 section 在最终 prompt 中的顺序。

`PromptComposer` 在初始化时会按四步执行：

1. `Filter`：用当前环境快照去求值每个 section 的条件谓词。返回 `False` 的 section 在文件读取前就被丢弃。例如 `main-git-workflow.md` 只有在 `in_git_repo` 为真时才会被加载。
2. `Sort`：按优先级升序排列保留下来的 section。较低优先级值会排在更前面，使身份与角色规则先于环境上下文出现。
3. `Load`：读取各个 markdown 模板，剥离 frontmatter，并解析 `${VAR}` 占位符。变量解析通过集中式名称注册表完成，使模板文案与真实工具名称解耦。
4. `Join`：把这些 section 串接到核心角色文本和动态环境块后面，形成最终的 system prompt。

默认的 action mode agent 会跨五个功能层注册这些模块：

- `Core Identity`：人格设定与不可协商约束。
- `Tool Definitions`：工具使用与代码质量规则。
- `Safety & Rules`：Git 约定、任务跟踪等按条件启用的策略。
- `Provider-Specific Guidance`：不同模型供应商特有的行为提示。
- `Dynamic Context`：会话级元数据。

而 thinking mode 故意加载更少的 section，特别是省略工具使用指导，避免在“无工具思考阶段”就被 prompt 诱导去提前行动。

### 模式专用模板与变量替换

不同推理阶段需要根本不同的提示词。

- Normal / Action mode：加载五层完整模板。
- Thinking mode：只加载少量专用 section，例如可用工具感知、subagent 提示、代码引用规范、输出格式规则。
- Planning mode：使用一个独立的、针对只读探索优化的模板。

这种差异由工厂函数控制：`create_composer(templates_dir, mode="system/main")` 会返回完整 action composer，而 `mode="system/thinking"` 则返回最小化 thinking composer。

模板内部通过 `PromptRenderer` 在渲染时解析 `${VAR}` 占位符。所有变量名都指向一个统一的 `PromptVariables` 注册表，例如 `${EDIT_TOOL.name}` 最终会解析为真实工具名 `edit_file`。这种间接层的价值在于：以后如果重命名某个工具，只需要改一个注册项，而不用在所有模板里全局替换。

### 按供应商条件加载的 section

不同 LLM 供应商在能力与接口上差异很大。Anthropic 支持 `tool_use` 内容块与 extended thinking；OpenAI 使用 function calling 与结构化输出；Fireworks 之类的推理供应商又可能有不同的上下文长度限制。如果不给 prompt 注入供应商特定说明，agent 很容易在 prompt 中引用自己实际上没有的能力。

OpenDev 的做法是：在运行时环境里读取 `model_provider` 字段，并基于它加载互斥的 provider-specific section。也就是说，每个 prompt 最多只会包含一个 OpenAI、Anthropic 或 Fireworks 的专属 section；未知 provider 则优雅降级，不加载任何供应商提示。

### 面向 provider 的 prompt caching

对支持输入缓存的 provider，例如 Anthropic，`PromptComposer` 提供 `compose_two_part()` 方法，把最终 prompt 拆成 stable part 和 dynamic part。每个 section 都可以用 `cacheable` 标记是否可缓存，默认值为 `true`。

- 可缓存部分：基础说明、工具描述、安全策略。
- 不可缓存部分：环境元数据、会话级上下文。

Anthropic 适配器会把这两部分组织成双元素内容数组：稳定块带 `cache_control: {"type": "ephemeral"}`，动态块则不带。由于 system prompt 会在每次调用时都被重新发送，而其中 80% 到 90% 往往是稳定内容，因此缓存这部分可以显著降低多轮会话的输入 token 成本。对不支持这一机制的 provider，系统仍然返回完整拼接后的单字符串，只是失去缓存收益，不改变行为。

### 双层回退

如果某个 section 文件缺失，composer 会跳过它并继续启动，因此 agent 只是拿到略微缩减的 prompt，而不会因为单文件丢失而完全无法运行。若整个模块化 prompt 组装失败，例如模板目录不存在，则构建器会退回到单体 core template。这样可以在“部分部署不完整”的条件下仍保证 agent 可启动。

## 12.3.2 工具结果优化

原始工具输出占用的 token，通常远高于它真正提供的信息价值。一次 `read_file` 可能返回两三千 token 的源代码；一次目录枚举可能列出上百项；一次测试命令可能吐出成千上万行结果。如果不加控制，冗长的工具结果会在几轮之内挤满上下文窗口，把用户问题和系统指令挤出去。工具结果优化的目标，就是在不破坏语义的前提下，把这些原始结果压缩成更紧凑的表示，再放入对话历史。

### 按工具类型做摘要

每个工具结果都会经过专门 summarizer，按工具类型应用不同压缩策略，通常把结果压缩到 50 到 200 个字符：

- 文件读取：用元数据替代正文，例如“已读取文件（142 行，4831 字符）”。完整内容仍然可以通过再次读取获取。
- 搜索结果：报告匹配数量，而不是把每一条命中内容都塞进上下文。
- 目录列表：折叠成项目数，例如“已列出目录（47 项）”。
- 命令执行：短输出直接保留；长输出则改为“执行完成（312 行输出）”。
- 错误信息：截断到 200 字符，并加上分类前缀，例如“错误：FileNotFoundError: ...”。

这种做法保证 agent 能知道“发生了什么”，却不需要一直背着“完整输出全文”。

### 大输出卸载到 scratch 文件

当工具输出超过 8000 个字符时，仅靠摘要仍然不够。OpenDev 会直接把完整输出写入会话专属 scratch 目录，例如 `~/.opendev/scratch/<session_id>/`，然后在对话里只保留一个约 500 字符的预览和一个路径提示：

“输出已卸载：2341 行，48203 字符 -> <path>。如需完整内容，请使用 `read_file` 查看。”

这样就形成了自然的分层：

- 对话中保留足够的预览，帮助 agent 判断内容大致是什么。
- 需要细节时，agent 再按需读取完整输出。

### 面向 agent 能力的截断提示

当输出被卸载到 scratch 文件时，OpenDev 还会根据当前 agent 的能力给出不同恢复建议：

- 如果 agent 有 `spawn_subagent`，系统会建议它委托一个 Code Explorer subagent 去逐步处理完整输出。
- 如果当前 agent 本身没有 subagent 权限，则系统提示它改用 `search` 的 `offset/limit` 参数分块处理。

这种 agent-aware 提示可以避免常见失败模式，例如一个 subagent 明明没有再生成 subagent 的能力，却被鼓励走一条根本不存在的恢复路径。

### 与 compaction 的协同

工具结果摘要与大输出卸载是互补关系。进入对话历史时，它们先做第一轮压缩；等真正触发 compaction 时，compactor 可以优先复用这些预计算的 summary，而不需要再次处理原始结果。这使得即使进入紧急上下文压缩阶段，喂给 compaction LLM 的输入也已经被显著瘦身，从而提升压缩速度与质量。

### 设计演化

早期版本无论工具输出多长，都会完整保留在对话历史里。一次长测试运行就可能吞掉三万 token 的上下文。引入按工具类型摘要后，大部分结果可以从几千 token 压缩到百 token 以内；再加上 8000 字符阈值的 scratch 卸载，系统把典型会话长度从 15 到 20 轮，延长到 30 到 40 轮而无需紧急压缩。

## 12.3.3 面向有界思考的双记忆架构

thinking 阶段需要会话上下文来进行策略推理，但完整对话历史可能增长到数十万 token。若把全部历史都给 thinking model，不仅会超出上下文窗口，还会把预算浪费在陈旧细节上；如果只给最近几条消息，又会失去整体目标与策略背景。OpenDev 用一套受认知科学启发的“双记忆架构”来解决这个张力：把长程压缩信息和短程高细节信息分开处理。

### 情景记忆（Episodic Memory）

系统会周期性地对完整对话历史生成 LLM 摘要，用来保留长程策略信息：已经做出的关键决策、总目标、重要发现、核心文件路径等。这个 summarizer 被要求保留可执行标识符，例如文件路径、函数名、变量名、错误码，同时省略冗长工具输出和重复交流。

情景记忆不是每一轮都重新生成，而是每新增 5 条消息后再刷新一次。这样做有两个原因：

- 摊薄摘要调用成本。
- 避免“摘要的摘要”造成的漂移。

OpenDev 不会反复压缩上一轮摘要，而是定期从完整历史重新生成一次新的情景记忆快照。

### 工作记忆（Working Memory）

最近若干轮消息会原样保留，默认是最近 6 组往返对话。这里包含了最具操作意义的细节：刚读过的文件内容、最新错误信息、精确行号、最近 tool call 的执行结果。这些细节如果被摘要化，恰恰会破坏下一步决策最需要的部分。

### 二者联合注入

在每次 thinking LLM 调用之前，系统会把三部分拼在一起：

1. 情景记忆摘要，提供“大局”。
2. 工作记忆消息，提供“当前操作细节”。
3. 当前用户请求。

这很像认知架构中的情景记忆与工作记忆分工：前者保留经验的 gist，后者保留最近且高精度的操作信息。由于情景记忆长度固定上限大约 500 字符，工作记忆窗口也固定，因此 thinking 阶段的 token 开销始终可控，不会随着会话无限增长。

### 设计演化

纯摘要方案会丢失关键标识符，导致 agent 指向错误文件或错误函数名；只保留最近消息又会让 agent 在十轮之后逐渐忘记用户原始目标。双记忆架构正是用来同时避免这两种失败模式。与此同时，OpenDev 也发现增量式“摘要上一轮摘要”会不断累积失真，因此才转向周期性从完整历史再生情景记忆。

## 12.3.4 上下文感知型系统提醒

长会话里，模型注意力会逐渐偏离最初 system prompt 中的约束，最终表现为一系列“静默失败”：任务没做完就宣布成功、遇错不重试、探索进入死循环等。系统提醒（system reminders）就是为了解决这个问题。

典型场景是：agent 在最初几轮还会乖乖遵守“改代码后必须跑测试”的系统规则，但经历十几二十次工具调用后，这条规则虽然仍在 prompt 里，却不再被模型真正关注。相同现象也会出现在 todo 执行、错误恢复和完成判定上。根源很简单：system prompt 位于对话最开头，而模型在长上下文中会把注意力逐步转向最近消息。规则仍然存在，但影响力随着距离衰减。

OpenDev 的解决方式不是周期性重发整份 system prompt，而是在真正要出错之前，注入一条短小、单用途、极高时效性的提醒。每条 reminder 都以 `role: user` 的形式放在下一次 LLM 调用前的最高近因位置，只做一件事，就是在关键决策点把模型拉回正确轨道。

### 事件检测器

提醒层在每一轮工具执行结束后，会检查多种条件是否成立。典型事件包括：

- 工具失败且未发起重试。
- 读文件行为连续 5 次以上，进入探索螺旋。
- 被拒绝的工具调用再次尝试。
- 仍有未完成 todo 却试图结束任务。
- todo 已全部完成却仍在无意义继续。
- plan 已批准但未实际执行。
- subagent 结果尚未处理。
- completion message 为空。

不同事件对应不同提醒模板，模板大致分为：阶段控制、任务生命周期、todo 约束、错误恢复、行为纠偏、JSON 重试等类别。

### 模板解析

所有 reminder 文本都存放在外部文本文件里，而不是硬编码在源代码中。短模板集中在 `reminders.md` 中，以命名 section 存放；更长模板则拆成单独 `.txt` 文件。`get_reminder(name, **kwargs)` 会在首次调用时把这些模板解析进模块级缓存，并使用 `str.format()` 替换占位符。

这种设计的好处是：提醒策略可审计、可编辑、可替换，不用改 Python 逻辑。

### 护栏计数器

提醒若每轮都注入，很快就会从“帮助模型”变成“模型学会忽略的噪声”。因此 OpenDev 对不同 reminder 设置了不同计数上限：

- incomplete todo 提醒最多触发两次。
- error recovery 提醒最多触发三次。
- plan approved、all todos complete、completion summary 这类提醒只触发一次。

若模型在有限次数的提醒后仍无改进，系统不会永远循环注入，而是接受当前状态继续执行或交给用户判断。

### 为什么用 `role: user`

实验发现，把提醒作为 `role: system` 注入时，效果明显不如 `role: user`。原因并不复杂：在 40 轮对话之后，再来一条 system message，模型往往把它当成背景说明；而 user message 出现在对话的最新位置，更像是一件“刚刚发生、必须响应”的事。因此，OpenDev 把 reminder 设计成高近因的 user-role 消息，以最大化它对下一步决策的影响。

### 优雅降级与设计演化

如果某条 reminder 模板缺失，系统依然还有初始 system prompt 可用，因此 reminder 只是强化而非必需条件。早期版本完全依赖 system prompt，在 30 次以上工具调用的长会话里，经常出现提前完成、探索循环、错误不恢复等注意力衰减问题。引入 just-in-time reminder 后，这些失败模式显著减少。后来系统又从通用的“请重试”提醒，演化为按错误类型分类的细粒度提醒，因为“重读文件再试”显然比“再试一次”更可执行。

## 12.3.5 注入上下文的错误恢复

工具调用失败时，原始错误消息会作为 tool result 进入对话历史。但仅有错误文本，往往只会让 agent 道歉，而不会自发采取正确恢复步骤。OpenDev 的做法是：把错误恢复指导本身也注入到上下文中，让错误消息变成可执行的恢复方案。

这个机制包含四步：

1. 按错误模式将失败归类到六种典型类别之一：权限错误、文件不存在、编辑不匹配、语法错误、限流、超时。
2. 从集中式模板库中取出对应恢复模板。
3. 用当前上下文细节填充模板，例如失败文件路径、冲突内容、具体错误文本。
4. 在下一次 LLM 调用前，把这段恢复提示作为最高近因的 system message 注入对话。

例如 edit mismatch 类错误会得到这样的指导：“文件在你上次读取后已经发生变化；请先重新读取最新内容，再基于当前内容重试编辑。”这显然比“请修复问题后重试”更具体，也更容易让模型按正确路径恢复。

每个错误序列默认只有 3 次恢复提醒预算，防止陷入无限重试。如果三次恢复都失败，系统会接受失败结果，允许 agent 继续、降级、或请求用户介入。由于模板以外部文本存储，因此新增错误类型往往只需添加模板，而不用修改主逻辑。

## 12.3.6 自适应上下文压缩

ReAct 循环运行时，工具观察结果，例如文件内容和命令输出，会迅速堆积并吞掉上下文窗口，往往在几轮之后就占掉 70% 到 80% 的 token 预算。很多系统只在 95% 到 99% 时才触发一次“紧急压缩”，把对话历史做一次损失很大的摘要。这种二元式做法的问题是：触发太晚、信息损失太大、后续多次压缩还会不断累积误差。

OpenDev 采用 Adaptive Context Compaction（ACC），在每一轮 ReAct 迭代开始时都监控上下文压力，并根据不同阈值逐步应用越来越激进的压缩策略，而不是等到窗口几乎爆满才处理。

### 五阶段压缩流水线

ACC 采用五个递进阶段：

- Stage 1，70%：仅记录警告，不做实际压缩。
- Stage 2，80%：对较旧的工具结果做 observation masking，用紧凑引用替换原始输出。
- Stage 2.5，85%：执行 fast pruning，对超出保护窗口的旧工具结果直接替换成 `[pruned]`。
- Stage 3，90%：进入 aggressive masking，只保留最近极少量工具输出为完整状态，其余都遮罩。
- Stage 4，99%：触发 full compaction，把完整历史序列化到 scratch 文件，并用 LLM 对对话中段做摘要，保留最近消息原文。

这种 staged compaction 的核心思想是：尽量先用廉价、可逆、局部的方式释放上下文，而不是过早进入代价高、损失大的全量摘要。

### Artifact Index

ACC 还维护一份 Artifact Index，记录本轮会话中所有被触及的文件与操作，包括读取、创建、修改、删除。这个索引会被一并写入 compaction summary 中，因此即使上下文被压缩，agent 仍能记得自己动过哪些文件。历史归档路径也会被显式写进摘要，例如：

“完整会话历史已归档至 <path>。如需细节，请使用 `read_file` 恢复。”

这使得 compaction 在设计上尽量接近“非破坏性压缩”：细节被移出主上下文，但并未永久丢失。

### 自适应记忆（ACE）

除了压缩，OpenDev 还通过 Agentic Context Engineering（ACE）子系统，把多次会话中积累的经验固化成一个 playbook。playbook 由一组自然语言 bullets 组成，每条 bullet 都带有效性计数（helpful、harmful、neutral）以及创建时间。它的维护流程大致分四步：

1. `BulletSelector` 按效果、时间衰减与语义相似度打分，选出最相关策略注入当前 system prompt。
2. 每 5 条消息触发一次 `Reflector`，对最近积累经验做反思，输出推理链、错误识别、根因分析与正确做法。
3. `Curator` 读取 reflection，生成对 playbook 的修改计划，例如新增、更新、重标记、删除条目。
4. 修改结果作为 `DeltaBatch` 应用到 playbook，并持久化到会话级 JSON 文件。

因此，ACC 管理的是“当前上下文怎么活下去”，而 ACE 管理的是“长期经验怎么留下来”。

## 12.3.7 上下文检索与组装流水线

对于编码智能体来说，context retrieval 是最关键的能力之一。下游一切编辑、测试、规划动作的质量，都受限于 agent 是否先找到正确代码。传统 RAG 是静态、一次性的：嵌入查询、取 top-k 文档、再生成。但代码库不同于一般文档语料，一个用户请求常常需要同时跨越 handler、schema、测试文件和配置模块。OpenDev 因此采用四层递进式检索与上下文组装流水线，把简单查找、多步探索、上下文拼装与预算优化串成一体。

### Layer 1：基于锚点的检索工具选择

OpenDev 提供五类核心检索工具：

- `read_file`
- `list_files`
- `text_search`
- `find_symbol`
- `ast_search`

关键不在于“有哪些工具”，而在于“agent 如何知道该用哪一个”。朴素 agent 往往逢问题就用全文搜索，这类似传统 RAG 的“全部都检索”反模式。OpenDev 鼓励 agent 先识别查询中的最强锚点，再匹配到相应工具：

- 符号名，例如 `AuthController.validate`，走 `find_symbol`。
- 字符串字面量和错误消息，走 `text_search`。
- 结构模式，例如“所有检查 `is_admin` 的 Python if 语句”，走 `ast_search`。
- 文件路径约定，则走 `list_files`。

这相当于在做“检索前先判断自己到底需要哪类信息”，从而减少低精度、大噪声搜索。

### Layer 2：通过 Code Explorer 进行多步智能检索

当主 agent 不知道该看哪个文件，或者需要沿着依赖关系逐步深入时，就会把问题委托给 Code Explorer subagent。这个 subagent 在独立上下文窗口中运行，只拿到上述五类只读检索工具，因此可以自主进行多步探索：

- 先用 `find_symbol` 找定义。
- 再 `read_file` 理解上下文。
- 再 `text_search` 追踪依赖的使用位置。

它的关键价值不是“多用几个工具”，而是把检索与推理交织起来：当前一步找到了什么，就决定下一步看什么。更重要的是，中间产生的大量搜索结果都被限制在 subagent 自己的上下文里，主 agent 最终只收到蒸馏后的摘要，从而避免检索本身吞噬主上下文。

### Layer 3：上下文组装

找到代码还不够，模型还需要行为指令、经验记忆和会话历史。`ContextPicker` 在每次模型调用前，会按固定顺序组装六类上下文源：

1. 注入所选 playbook 策略后的 system prompt。
2. 项目级与用户级持久规则。
3. 用户提供的 `@file` 引用与图片块。
4. 从 `SessionManager` 取回的对话历史。
5. 在决策点注入的 system reminders。
6. 当前用户请求。

每个上下文块都被包装为 `ContextPiece`，携带来源、优先级与 token 成本等元数据，以便后续压缩阶段可以做更聪明的保留决策。

最终，组装好的 message list 会经过 `ValidatedMessageList` 检查结构完整性。例如，凡是带 tool calls 的 assistant 消息，后面必须跟上对应的 tool results；若结构损坏，系统会尝试自动修复，而不是直接失败。

### Layer 4：上下文优化

组装好的上下文会进入前面介绍的 staged compaction 流程。观察结果会随着时间经历三个状态：

- active：最近、完整保留。
- faded：达到 80% 阈值后可被遮罩。
- archived：序列化到磁盘并以引用形式保留。

此外，OpenDev 会用上一次 API 返回的真实 `prompt_tokens` 作为校准锚点，而不是完全依赖本地估算，从而修正 provider 侧额外注入内容造成的偏差。这使最后提交给 LLM 的 payload，不仅符合模型的上下文上限，而且更接近“保留了真正对决策最重要的信息”。

### 端到端流

这四层形成了一个渐进升级的检索流水线：

- 简单查询在 Layer 1 用一次工具调用就解决。
- 复杂探索升级到 Layer 2 的多步 agentic search。
- 所有结果汇总到 Layer 3 统一组装。
- Layer 4 再负责把最终上下文压到预算以内。

这种设计让“读取单个文件”这类简单任务几乎没有额外成本，而“搞清楚整个认证系统怎么工作的”这类开放式任务，则可以充分使用多步检索与上下文编排能力，而不会把检索成本直接溢出到主 agent 的上下文里。

## 小结

上下文工程层真正决定了模型“看见什么”，而不是模型“会不会调用某个更强参数的大模型”。OpenDev 在这一层做的工作，包括动态提示词组装、工具结果压缩、双记忆架构、即时提醒、模板化错误恢复、自适应上下文压缩，以及从检索到组装的完整流水线。它们共同回答了一个比“模型多强”更关键的问题：在一个长时间运行、工具丰富、上下文有限的终端智能体系统里，什么信息应该被保留，什么应该被延后，什么应该被压缩，什么必须在决策前重新推到模型面前。
