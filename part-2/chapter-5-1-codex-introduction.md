# 6.1 OpenAI Codex 简介与演进脉络

本节聚焦三个问题：

1. Codex 最早解决了什么问题？
2. 为什么今天讨论 Codex 不能只停留在“代码补全”？
3. 从 Codex 到 Agent Loop / Harness Engineering，工程范式发生了什么变化？

---

## 3.1.1 Codex 的历史定位：代码大模型的起点之一

OpenAI Codex 是 GPT-3 系列在代码语料上强化后的关键里程碑。它把“自然语言理解能力”与“代码生成能力”结合起来，第一次让大规模开发者看到：

- 模型不仅能补全下一行，还能根据注释生成函数级逻辑。
- 模型不仅能“写”，还能“解释”“转换”“修复”部分代码。
- 人机协作可以从“查文档 + 手写模板”转向“描述需求 + 迭代验收”。

这也是 GitHub Copilot 得以快速普及的基础：它把 AI 能力嵌入编辑器工作流，而不是独立在聊天窗口中。

---

## 3.1.2 从 Codex 到现代 GPT 编程能力

虽然“Codex”作为独立产品名称出现频率下降，但其核心思想并未消失，而是演进到新一代模型的代码能力体系里：

- 更强的跨文件理解（不仅看当前函数）。
- 更稳定的多步推理（先分析后改动）。
- 更完整的工具调用链（读文件、执行命令、跑测试、再修复）。

因此，今天讲 Codex，不应只把它当作“过去式模型”，而应把它看作现代 AI 编程代理能力的源头之一。

---

## 3.1.3 从 Prompt Engineering 到 Harness Engineering

早期实践中，很多人把提升效果的重点放在 Prompt Engineering（提示词工程）。这当然重要，但在真实项目里还不够。

复杂任务要稳定完成，至少需要四类条件同时满足：

- **任务清晰**：输入、输出、边界、验收标准明确。
- **环境可复现**：依赖版本、命令、权限和数据一致。
- **质量可度量**：测试、lint、类型检查能程序化判断结果。
- **失败可迭代**：错误信息可回流给模型继续修复。

这就是 Harness Engineering 的核心：
**不是只优化“怎么问”，而是构建“模型如何被工程化约束并持续交付”的执行支架。**

---

## 3.1.4 Codex Agent Loop：把一次生成变成收敛循环

从 `Unrolling the Codex Agent Loop` 的视角，现代编码代理可抽象为五步闭环：

1. **Observe（观察）**：读取代码、配置、报错、任务约束。
2. **Plan（规划）**：拆解任务、选择改动路径、识别风险点。
3. **Act（执行）**：做最小必要改动并记录变更原因。
4. **Check（验证）**：运行测试/检查命令获得客观反馈。
5. **Refine（迭代）**：基于失败信息修复并重复验证。

关键认知：
- 单轮输出不完美是常态；
- 能不能高质量交付，取决于“循环是否可收敛”。

---

## 3.1.5 对团队落地的实际启发

如果你在团队里使用 Codex/Agent 系统，这三条原则最实用：

- **先定义验收，再让模型动手**：没有验收标准，模型就只能“猜对”。
- **先跑检查，再谈质量**：测试结果比“回答看起来很像正确”更可靠。
- **先可追溯，再追求速度**：每轮改动的命令、日志、理由都应可追踪。

把这三条执行好，Codex 的价值就会从“写得快”升级为“交付稳”。

---

## 3.1.6 小结

Codex 的意义不止于“它能写代码”，更在于它开启了一条清晰演进路径：

**代码生成能力（Codex） → 工具化协作能力（Agent） → 可验证交付能力（Harness + Loop）**。

这条路径决定了 AI 编程能否真正进入团队的主流程，而不仅是个人效率插件。
