# 5.13: Lessons from Building Claude Code — 像 Agent 一样思考

> *One of the hardest parts of building an agent harness is constructing its action space.*

## 一、构建 Agent 动作空间的挑战

构建 Agent 工具的核心挑战在于：如何设计 Agent 可用的工具集合？Claude 通过 Tool Calling 运作，但可以通过多种方式构建工具——如 Bash、Skills，以及最近的代码执行。

### 1.1 设计工具的核心问题

- 只需要一个工具（如代码执行或 Bash）吗？
- 还是需要 50 个工具，每个对应一个用例？

### 1.2 像 Agent 一样思考的方法

**把自己想象成 Agent**：想象自己被给定一道困难的数学题。你需要什么工具来解决问题？

- 纸笔是最基本的，但受限于手动计算能力
- 计算器更好，但你需要知道如何操作高级功能
- 计算机最快最强大，但你需要知道如何编写和执行代码

这是一个设计 Agent 工具的有用框架。你想给它与自身能力相匹配的工具。但你如何知道这些能力是什么？你需要**关注它、阅读它的输出、进行实验——学会像 Agent 一样看待事物**。

## 二、Claude Code 的工具设计经验

### 2.1 改进 Elicitation — AskUserQuestion 工具

构建 AskUserQuestion 工具的目标是提高 Claude 提问（称为 elicitation）的能力。

虽然 Claude 可以用纯文本提问，但团队发现回答这些问题感觉需要不必要的时间。如何降低这种摩擦，增加用户与 Claude 之间的沟通带宽？

**尝试 #1 — 修改 ExitPlanTool**

第一个尝试是在 ExitPlanTool 中添加一个参数，包含问题数组以及计划。这是实现最简单的，但让 Claude 感到困惑，因为同时要求提供计划和关于计划的问题。如果用户的答案与计划冲突怎么办？Claude 需要两次调用 ExitPlanTool 吗？需要另一种方法。

**尝试 #2 — 更改输出格式**

接下来尝试修改 Claude 的输出指令，使用稍微修改的 Markdown 格式来提问。例如，可以要求它输出带括号中选项的点问题列表。然后可以解析并将该问题格式化为 UI 供用户使用。

虽然这是可以做出的最通用更改，Claude 甚至似乎可以很好地输出这种格式，但不能保证。Claude 会附加额外句子、省略选项或使用不同格式。

**尝试 #3 — AskUserQuestion 工具**

最终设计是创建一个 Claude 可以在任何时候调用的工具，但在 Plan Mode 中特别提示使用它。当工具触发时，会显示一个模态框来显示问题，并阻止 Agent 循环直到用户回答。

这个工具允许我们提示 Claude 提供结构化输出，并帮助确保它为用户提供多个选项。它还为用户提供了一些组合此功能的方式，例如在 Agent SDK 中调用它或在 Skills 中引用它。

最重要的是，Claude 似乎喜欢调用这个工具，团队发现它的输出效果很好。即使设计最好的工具，如果 Claude 不知道如何调用它，也不会起作用。

### 2.2 随着能力更新 — Tasks 与 Todos

Claude Code 最初发布时，团队意识到模型需要一个 Todo 列表来保持正轨。Todo 可以在开始时写入，并在模型完成工作时勾选。为此，给了 Claude TodoWrite 工具来写入或更新 Todo 并显示给用户。

但即使如此，团队经常看到 Claude 忘记它需要做什么。为了适应这一点，每 5 轮插入系统提醒，提醒 Claude 它的目标。

但随着模型改进，它们不仅不需要被提醒 Todo 列表，而且可能会发现它有限制。被发送 Todo 列表提醒让 Claude 认为它必须坚持列表，而不是修改它。团队还看到 Opus 4.5 在使用 Subagent 方面做得更好，但 Subagent 如何协调共享的 Todo 列表？

看到这一点后，团队用 Task 工具替换了 TodoWrite 工具。虽然 Todo 是关于让模型保持正轨，但 Tasks 更多是关于帮助 Agent 相互通信。Tasks 可以包括依赖项、在 Subagent 之间共享更新，模型可以更改和删除它们。

**关键洞察**：随着模型能力的增长，你曾经需要的工具现在可能限制了它们。不断重新审视关于需要哪些工具的先前假设也很重要。

### 2.3 设计搜索接口

对 Claude 特别重要的一组工具是用于构建其自身上下文的搜索工具。

当 Claude Code 首次发布时，团队使用 RAG 向量数据库为 Claude 查找上下文。虽然 RAG 强大且快速，但它需要索引和设置，在许多不同环境中可能很脆弱。更重要的是，Claude 是被动获得这个上下文，而不是自己找到它。

但是，如果 Claude 可以在网上搜索，为什么不能在代码库中搜索？通过给 Claude Grep 工具，可以让它搜索文件并自己构建上下文。

这是团队随着 Claude 变聪明而看到的一个模式：如果给它正确的工具，它在构建自己的上下文方面变得越来越擅长。

**渐进式披露**是团队用来添加新功能而无需添加工具的常见技术。当引入 Agent Skills 时，正式化了渐进式披露的想法，允许 Agent 通过探索逐步发现相关上下文。

Claude 可以读取技能文件，这些文件可以递归引用其他文件，模型可以读取。实际上，技能的常见用途是向 Claude 添加更多搜索能力，例如给它有关如何使用 API 或查询数据库的指令。

在一年多的时间里，Claude 从不太能够构建自己的上下文，到能够跨多个文件层进行嵌套搜索，找到它需要的确切上下文。

### 2.4 渐进式披露 — Claude Code Guide Agent

Claude Code 目前有大约 20 个工具，团队不断问自己是否需要所有这些。添加工具的门槛很高，因为这给模型多了一个需要考虑的选择。

例如，团队注意到 Claude 不知道足够多关于如何使用 Claude Code 本身。如果你问它如何添加 MCP 或什么是斜杠命令，它无法回答。

他们本可以将所有这些信息放入 System Prompt，但考虑到用户很少问这个问题，这会增加上下文腐败并干扰 Claude Code 的主要工作：编写代码。

相反，他们尝试了一种渐进式披露形式。他们给 Claude 一个指向其文档的链接，然后它可以加载来搜索更多信息。这有效了，但团队发现当真正只需要答案时，Claude 会加载很多结果到上下文中。

所以他们构建了 Claude Code Guide Subagent，当用户询问关于 Claude Code 本身时，Claude 会被提示调用该子代理。该子代理有广泛的指令，告知它如何很好地搜索文档以及返回什么。

虽然这并不完美——当用户问如何设置 Claude Code 时，Claude 仍然可能会混淆——但它比过去好得多！他们能够在不添加工具的情况下为 Claude 的动作空间添加东西。

## 三、工具设计的核心原则

### 3.1 工具设计是艺术而非科学

如果你希望得到关于如何构建工具的严格规则，抱歉，这不是本指南。设计模型的工具既需要艺术也需要科学。它严重取决于你使用的模型、Agent 的目标及其运行环境。

### 3.2 实践建议

| 原则 | 说明 |
|------|------|
| **从小处开始** | 只提供必要的工具，避免过度设计 |
| **观察模型行为** | 关注模型如何使用工具，哪些有效哪些无效 |
| **渐进式披露** | 不要把所有信息放入 System Prompt，而是让模型通过工具发现 |
| **使用状态工具而非更换工具** | 如 Plan Mode，使用 EnterPlanMode/ExitPlanMode 而不是切换工具集 |
| **随着模型能力迭代** | 曾经需要的工具可能成为限制，定期重新评估 |

### 3.3 像设计师一样思考

## 四、案例总结

### AskUserQuestion 工具

- **问题**：用户回答 Claude 的提问太慢
- **解决**：创建专门工具，确保结构化输出和多个选项
- **关键**：工具要让模型愿意调用

### Tasks 工具

- **问题**：Todo 列表对改进的模型变成限制
- **解决**：替换为支持依赖和跨 Agent 共享的 Tasks
- **关键**：工具要随模型能力迭代

### 工具搜索

- **问题**：大量 MCP 工具增加成本，删除又破坏缓存
- **解决**：使用 defer_loading 存根 + 发现机制
- **关键**：渐进式披露，避免一次性暴露所有能力

### Claude Code Guide

- **问题**：模型不了解自身使用方法
- **解决**：构建专门 Subagent 而非扩充 System Prompt
- **关键**：缓存友好，按需加载

## 五、结语

设计 Agent 工具是一个需要不断实践、观察和迭代的过程。没有放之四海而皆准的规则——关键是要学会像 Agent 一样观察和思考。

> *Experiment often, read your outputs, try new things. See like an agent.*

**实践清单**：
- [ ] 为你的用例设计最小工具集
- [ ] 观察模型实际如何使用工具
- [ ] 使用渐进式披露而非一次性暴露
- [ ] 设计工具时考虑缓存影响
- [ ] 随着模型能力迭代更新工具设计
