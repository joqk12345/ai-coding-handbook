# 5.4: Claude Code 创造者如何使用 Claude Code

> 参考线索：<https://x.com/bcherny/status/2007179832300581177>
>
> 引用说明：本章 2.4.7 的 13 个技巧来自 Boris Cherny 在 X 的原始帖子，中文表述为便于阅读做了整理与归纳，语义保持一致。

这一章不再讲“功能清单”，而是讲**工作方式**：把 Claude Code 当作一个持续协作的工程搭档，而不是一次性问答工具。核心目标是：减少上下文切换、缩短反馈回路、把复杂任务拆成可验证的小步。

---

## 2.4.1 先让 Claude 理解你的项目，再开始提需求

创造者工作流里最重要的一步通常不是“直接让它写代码”，而是先对齐上下文：

- 先让 Claude 读项目结构、构建命令、测试入口。
- 把关键约束说清楚（语言版本、代码风格、不可改动模块）。
- 把“验收标准”提前说出来（通过哪些测试、输出哪些文件）。

这样做的收益是：后续每轮交互都更聚焦，返工更少。

**示例提示词：**

```text
先不要改代码。请先读取项目并总结：
1) 构建与测试命令
2) 关键目录职责
3) 我这个任务的潜在影响范围
约束：不要修改 payment/ 目录；Node 版本固定 20。
```

## 2.4.2 小步快跑：一次只解决一个明确子问题

高质量协作通常遵循这个节奏：

1. 先让 Claude 给出计划（改哪些文件、为什么）。
2. 只批准它做一个小范围修改。
3. 立刻运行测试或静态检查。
4. 根据结果进入下一步。

这比“直接大改整个仓库”更稳健，也更容易定位问题来源。

**示例提示词：**

```text
目标：修复登录超时 bug。
请先给我一个 3 步计划，不要立即改代码。
我确认后，你只执行第 1 步并展示 diff。
```

## 2.4.3 让工具调用透明化，而不是黑盒执行

创造者思路强调：**你要看到 AI 做了什么，而不是只看最终答案**。

实践上可以这样做：

- 要求它在每一步说明将执行的命令与预期结果。
- 先看 diff，再决定是否继续。
- 对风险操作（删除、迁移、大规模重构）设置额外确认点。

这样可以把“AI 试错成本”控制在可接受范围内。

**示例提示词：**

```text
执行前先列出你将运行的命令，并说明每个命令的目的。
完成修改后先给我 diff 摘要，再运行测试。
如果涉及删除文件或 schema 变更，先停下来让我确认。
```

## 2.4.4 把 Claude 当“结对程序员”，不是“外包程序员”

一个常见误区是：把需求一次性丢给 AI，等它“交付成品”。

更有效的方法是结对式协作：

- 你负责方向、取舍和验收。
- Claude 负责执行、探索和候选方案。
- 双方通过多轮短反馈快速收敛。

这会显著提升结果质量，尤其在架构权衡和边界条件处理上。

**示例对话节奏：**

- 你：先给我两个实现方案，重点比较复杂度和可维护性。  
- Claude：给出 A/B 方案 + 风险。  
- 你：选 A，但要兼容现有 API，不要引入新依赖。  
- Claude：按约束落地实现并补充回归检查。

## 2.4.5 先写可验证的目标，再追求“聪明”

创造者视角里，Claude Code 的价值首先来自**可验证产出**：

- 能跑的命令
- 可审阅的 diff
- 可复现的测试结果
- 清晰的失败原因

当这些基础闭环稳定后，再逐步引入更复杂的自动化（如多步骤任务链、跨文件重构）。

**示例验收标准：**

```text
验收条件：
- `npm test` 全部通过
- 新增用例覆盖登录超时分支
- PR 描述包含根因、修复点、回归风险
```

## 2.4.6 一个可直接复用的日常模板

你可以把下面这段作为每日协作起手式：

```text
你是我的结对工程师。请按以下流程执行：
1) 先复述目标与约束；
2) 给出最多 5 步的执行计划；
3) 每次只做一步并展示 diff；
4) 每一步后运行最小必要检查并汇报结果；
5) 任何高风险操作先暂停并请求确认。
```

这类模板的作用是把协作节奏固定下来，让每次会话都可预测、可审计。

## 2.4.7 Boris 的 13 个技巧（按原文语义整理）

基于参考文章《Claude Code 创始人 Boris Cherny 的 13 个使用技巧完整解析》，这里把 13 个技巧按“做法 + 适用场景”整理为可执行清单：

1. **并行运行多个 Claude 会话（例如 5 个）**  
   做法：同一时间让多个会话并行处理不同任务，并配合系统通知避免遗漏。  
   场景：你有多个互不阻塞的任务（调试、重构、文档、测试）并行推进。

2. **本地终端 + claude.ai/code 网页版协作**  
   做法：本地与云端会话互相切换，使用会话迁移能力（如推送到云端、再拉回本地）。  
   场景：需要在“本地资源访问能力”和“云端持续运行能力”之间切换。

3. **复杂任务优先使用更强模型（如 Opus + thinking）**  
   做法：用更强模型换取更少返工和更高一次成功率。  
   场景：架构改造、跨文件重构、复杂排障等高复杂度任务。

4. **团队共享并持续迭代 `CLAUDE.md`**  
   做法：把“应该怎么做/不要怎么做”沉淀到项目记忆，放进 Git 持续维护。  
   场景：团队协作项目，希望让经验可复用、可传承。

5. **Code Review 阶段反哺 `.claude` 规则**  
   做法：在评审里把发现的问题回写到 Claude 规则中（例如通过自动化流程写入）。  
   场景：希望把“发现问题”变成“预防问题”的团队闭环。

6. **多数任务先从 Plan Mode 开始**  
   做法：先讨论方案与步骤，再进入自动编辑执行。  
   场景：对方向正确性要求高、返工成本高的任务。

7. **用 Slash Commands 固化高频操作**  
   做法：把常用流程抽成 `/xxx` 命令，一键调用。  
   场景：PR 总结、发布说明、回归检查等重复流程。

8. **用 Subagents 处理专门子任务**  
   做法：把特定类型工作交给专门子智能体（如测试、文档、审查）。  
   场景：任务多线程、角色分工明确、需要并行专业处理。

9. **使用 PostToolUse Hook 自动格式化**  
   做法：在写入/编辑后自动触发格式化，减少手动整理。  
   场景：团队代码风格严格、频繁编辑文件。

10. **预设权限，而不是直接跳过权限系统**  
    做法：把安全命令预先放行，保留高风险操作确认。  
    场景：追求效率但不牺牲安全边界的团队。

11. **用 MCP 接入外部系统能力**  
    做法：连接 Slack、日志、数据库、监控等外部工具扩展 Agent 能力。  
    场景：任务需要跨系统取数、检索上下文、联动通知。

12. **长任务配置自动验证与监控机制**  
    做法：任务结束自动触发验证（或持续监控），避免“跑完无人验收”。  
    场景：长时间运行、夜间批处理、异步后台任务。

13. **给 Claude 明确“验证工作是否完成”的方式**  
    做法：提供可执行验证路径（测试命令、UI 回归、集成检查），形成反馈闭环。  
    场景：所有需要质量保障的任务；这是提升结果质量的关键技巧。

### 可直接复用的一段总提示词

```text
按 Boris 的 13 技巧执行本任务：
- 先进入 Plan Mode 给出步骤；
- 只在我确认后执行；
- 每步输出 diff 与执行命令；
- 完成后自动运行验证并汇报；
- 高风险操作必须暂停确认；
- 最后输出结构化复盘（根因/改动/验证/风险）。
```

## 2.4.8 本章小结

**原始引用链接**：<https://x.com/bcherny/status/2007179832300581177>


“创造者如何使用 Claude Code”的关键不在某个单一技巧，而在**工程化协作习惯**：

- 先对齐上下文
- 小步迭代
- 强调可验证输出
- 人类持续把关

当你这样使用 Claude Code，它就不只是“会写代码的聊天工具”，而是一个真正可落地的开发协作系统。
