# 6.3 Claude Code vs Codex: 权威对比指南

本文基于 Hesamation (@Hesamation) 的深度对比研究，从多个维度系统分析 Claude Code 与 OpenAI Codex 的差异。作者分别使用两款工具数月，并在相同任务（构建 RAG 流水线）上进行了详细对比测试。

> **阅读提示**: 本文约需 12 分钟阅读。如果你正在考虑每月投入 $200 订阅其中一款工具，这将是一次值得的时间投资。

## 1. Opus 4.6 vs GPT-5.3-Codex: 任务完成时间跨度

比较 Codex 与 Claude Code 的一个可靠方法是关注其底层旗舰模型及**任务完成时间跨度 (Task-Completion Time Horizon)**。

### 什么是任务完成时间跨度？

这一指标衡量的是模型能够可靠完成的任务时长（以人类专家完成时间衡量）的可靠度水平。例如，一个"2 小时时间跨度 @ 50%"的模型意味着：给它一个需要熟练人类 2 小时完成的任务，AI 大约有一半的成功率。

### 关键对比数据

| 模型 | 50% 成功率的时间跨度 | 80% 成功率的时间跨度 |
|------|---------------------|---------------------|
| **Opus 4.6** (Claude Code) | **12 小时** | ~6 小时 |
| **GPT-5.3-Codex** | **5 小时 50 分钟** | ~4 小时 |

> 数据来源: 基于 Morph Lab 的综合评测，测试使用了 Claude Code 和 Codex 各自的适当脚手架。

### 关键洞察

1. **显著的差距**: 在 50% 成功率水平上，Opus 4.6 能处理的任务时长是 GPT-5.3-Codex 的两倍多
2. **差距在高可靠度下缩小**: 当要求 80% 成功率时，两款模型的差距明显缩小
3. **实际意义**: 这表明 Claude Code 在处理困难、复杂的长期任务方面具有显著优势，尽管这种优势可能不会直接转化为日常简单任务的体验差异

## 2. 速度对比: Claude Code 更快，但速度并非关键

Claude Code 以速度著称，确实比 Codex 更快。但**与 AI 编码助手协作是一个长期过程**。

### 速度 vs 质量的权衡

| 场景 | Claude Code | Codex |
|------|-------------|-------|
| 任务完成时间 | 更快 | 较慢 |
| 首次尝试准确性 | 高 | **更高** |
| 后续调试需求 | 可能更多 | 通常更少 |
| 实现速度 | 快速推进 | 稳步推进 |

### 关键洞察

> 如果一款工具能在一半时间内完成任务，但随后需要你花费 10 分钟调试；而另一款工具花费更多时间实现，但后续不需要太多干预——那额外的实现时间 100% 是值得的。

**注意**: 这并不是说 Claude Code 或 Codex 中哪一款会犯更多错误，而是一个在评估 AI 编码助手时需要牢记的一般性原则。

## 3. 任务类型决定表现

Codex 和 Claude Code 在不同编码任务上表现各异。在某一类任务（如 AI 工程）中，一款模型可能优于另一款；而在另一类任务（如 Web 开发）中，情况可能完全相反。

### 任务适用性对比（基于观察）

| 任务类型 | Claude Code | Codex | 说明 |
|---------|-------------|-------|------|
| **AI/ML 工程** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | Claude 在 RAG、模型调优任务上表现更稳健 |
| **Web 开发** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | Codex 在前端、全栈项目上首次尝试准确性更高 |
| **系统/底层编程** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 两者表现接近，需要更多测试验证 |
| **数据处理/脚本** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 两者都能很好地处理 |

> **研究空白**: 目前缺乏系统性的研究来全面比较两款模型在不同编码任务上的表现。这是一个值得关注的研究领域。

### 建议

在全面投入之前，理想情况下你应该在简单且可验证的设置中测试两款模型。但花费 $300-$400 同时订阅两款工具对大多数人来说并不现实。

**实用建议**: 先尝试 $20/月的入门版，在你相关的编程领域进行测试，最好是在几个可验证的任务上进行。

## 4. 产品历史与发展脉络

### Claude Code 的起源

Claude Code 最初是由 Anthropic 的 @bcherny 开发的一个 side project。他构建了一个能够与 Claude API 交互、读取文件和运行一些 bash 命令的终端原型。

- **内部采用**: 内部团队一半人在第 5 天就开始使用
- **正式发布**: 2025 年 2 月 24 日作为研究预览版发布，使用 Claude 3.7 Sonnet
- **普及过程**: 花了一些时间才被开发者广泛采用
- **扩展**: 后来发布了 VS Code 扩展

### OpenAI Codex 的演进

OpenAI 最初将原始 Codex 模型宣布为一个在 GitHub 代码上微调的 12B GPT-3 模型，后来成为首个 GitHub Copilot 的驱动力。

- **新 Codex**: 现在的 Codex 是一个全新的产品
- **Codex CLI 发布**: 2025 年 4 月 16 日作为终端 agent 首次发布
- **持续演进**: 发布以来一直在改进，使用更好的模型
- **最新模型**: GPT-5.3-Codex (2026 年 2 月 5 日发布)，被 OpenAI 描述为"第一个帮助创造自身的模型"

### 深度访谈推荐

@GergelyOrosz (The Pragmatic Engineer) 对 Claude Code 和 Codex 的开发者进行了两次非常有趣的深度访谈，涵盖了他们的技术栈、开发方式以及各自的起源。

- [How Codex is Built](https://newsletter.pragmaticengineer.com/p/how-codex-is-built)
- [How Claude Code is Built](https://newsletter.pragmaticengineer.com/p/how-claude-code-is-built)

## 5. 技术栈与模型规格

### Claude Code

| 特性 | 规格 |
|------|------|
| **开发语言** | TypeScript |
| **UI 框架** | React + Ink (终端 UI 渲染) |
| **分发方式** | 单个 Bun 可执行文件 |
| **备注** | Anthropic 于 2025 年 12 月收购 Bun |
| **上下文窗口** | 1M tokens |
| **主要模型** | Opus 4.6, Sonnet 4.6 |

### Codex CLI

| 特性 | 规格 |
|------|------|
| **开发语言** | Rust |
| **选择理由** | 性能、正确性、可移植性 |
| **TUI 库** | Ratatui |
| **备注** | OpenAI 聘请了 Ratatui 维护者加入团队 |
| **主要模型** | GPT-5.3-Codex |

### 技术栈观察

两款 CLI 工具本质上都是围绕其各自模型的薄包装层，通过 API 调用底层模型。值得注意的是：

- **Claude Code CLI**: 基于 TypeScript/React 构建，使用 Ink 进行终端渲染
- **Codex CLI**: 基于 Rust 构建，性能导向

作者观察到 Claude Code CLI 偶尔会出现一些"小故障"，而 Codex CLI 则没有这些问题——这可能是技术栈差异的副产品。但这些问题只是轻微烦人，不会影响实际编码体验。

## 6. 基准测试详解

### Token 经济: 显著差异

两款工具之间最大的性能差异不是准确性，而是 **token 效率**。

Morph Lab 对 Opus vs Codex 的综合评测显示了一个有趣的差距：

| 指标 | Claude Code | Codex | 倍数差异 |
|------|-------------|-------|----------|
| **相对 Token 消耗** | 基准 (3.2-4.2x) | 1x | 3.2-4.2x |
| **Figma 插件构建** | 6.2M tokens | 1.5M tokens | ~4.1x |

这意味着，**支付相同价格的 Claude Code 订阅费用，你更可能达到 token 上限**。

### 实际影响

- 高 token 消耗可能导致更快达到使用限制
- 对于大型项目或频繁使用者，这可能成为实际瓶颈
- 需要在选择时考虑你的使用模式和预算

## 7. 用户体验: "感觉"最重要

### 两款工具的个性差异

开发者通常这样描述两者的区别：

| 维度 | Claude Code | Codex |
|------|-------------|-------|
| **角色定位** | 为你工作的资深开发者 | 你分配任务后回来取结果的承包商 |
| **交互风格** | 强互动感，深度推理 | 直接、高效、任务导向 |
| **沟通方式** | 主动提问，展示推理，解释方法 | 首次尝试准确，实现速度稍慢 |
| **典型反馈** | "让我理解一下..." "我会这样做，因为..." | "任务完成。" "这是结果。" |

### 实际交互差异示例

**Claude Code 风格**:
```
我理解你想构建一个 RAG 管道。让我先澄清几个关键点：
1. 你希望使用哪种嵌入模型？
2. 对于分块策略，考虑到这是学术论文，我建议...

我的实施计划是...
```

**Codex 风格**:
```
我来为你实现 RAG 管道。

[直接开始编写代码]

完成。你可以运行 `pip install -r requirements.txt` 然后 `python main.py` 来测试。
```

### AGENTS.md 的影响

值得注意的是，当你在 AGENTS.md 中明确指定需求时，两款工具的行为差异会减小：

- 如果你指定"需要在实施前与我确认计划"，两款模型都会这样做
- 行为差异依然存在，但没有社区讨论的那么夸张

**结论**: 两款工具确实不同，但差异不如你在 X/Twitter 上听到的那么极端。

## 9. 案例研究: RAG 管道实现对比

### 实验设计

为了进行量化评估，作者选择了一个可以客观测量的任务：构建 RAG（检索增强生成）管道。相比创建登录页面这种主观任务，RAG 的答案准确性可以用数字衡量。

**任务描述**: 构建一个用于研究论文的 RAG Q&A 管道

**工作流程**:
1. 提取多篇论文的文本
2. 将内容分块成更小的片段
3. 将每个块嵌入到向量空间
4. 当用户提问时，找到与问题嵌入最接近的块嵌入
5. 检索原始形式的接近块（非其嵌入）
6. 使用该上下文回答用户问题

### 实验设置

**数据集**:
- 从 @huggingface 本周每日论文中选取 5 篇研究论文
- 创建 100 个问题的测试集，包含问题及标准答案

**指定给两款工具的要求**:
- 构建 Python RAG 管道
- 使用 `PyMuPDF` 处理所有 PDF
- 为此用例选择良好的分块策略
- 创建嵌入和持久本地向量索引（自选）
- 使用 `llama-3.1-8b-instant` 生成最终答案
- 如果未找到充分证据，不要幻觉，返回回退响应

**模型配置**:
- Claude Code: Opus 4.6，High effort
- Codex: GPT-5.3-Codex，High effort
- 两者均无 AGENTS.md

### 实现方式对比

#### 相似之处

两款工具在以下方面选择了相似的方案：

| 组件 | Claude Code | Codex | 说明 |
|------|-------------|-------|------|
| **嵌入模型** | all-MiniLM-L6-v2 | all-MiniLM-L6-v2 | 相同的轻量级嵌入模型 |
| **Top-K 检索** | k=5 | k=5 | 都选择返回 5 个最相关块 |
| **LLM 约束** | 限制仅使用提供的上下文 | 限制仅使用提供的上下文 | 都通过系统提示防止幻觉 |

#### 不同之处

**向量存储选择**:

| 特性 | Claude Code (ChromaDB) | Codex (FAISS) |
|------|------------------------|----------------|
| **类型** | 完整的向量数据库 | 低层级相似性搜索库 |
| **资源效率** | 较高资源占用 | 更节省内存 |
| **速度** | 良好 | 更快 |
| **复杂度** | 更简单的 API | 需要更多手动配置 |

**分块策略**:

| 特性 | Claude Code | Codex |
|------|-------------|-------|
| **方法** | 递归字符分割 | 句子级词分割 |
| **分割边界** | `\n\n` → `\n` → `.` → 空格 | 句子边界 → 词预算填充 |
| **目标大小** | 1000 字符，200 字符重叠 | 220 词，40 词重叠 |
| **优势** | 按结构层次分割 | 尊重句子边界，避免句内切割 |
| **潜在问题** | 可能切割句子 | 220 词对学术论文可能过小 |

**检索逻辑**:

| 特性 | Claude Code | Codex |
|------|-------------|-------|
| **距离度量** | L2 距离 | 内积 (余弦相似度) |
| **返回格式** | 原始 L2 距离值 | 余弦相似度分数 |

**置信度判断**:

| 特性 | Claude Code | Codex |
|------|-------------|-------|
| **策略** | 单阈值 + 平均距离检查 | 多标准三层级 |
| **阈值类型** | 最佳 L2 距离 > 1.2 = 不相关 | 强/中等/不足三个等级 |
| **额外检查** | 平均距离判断低置信度 vs 良好基础块 | 多维度综合评估 |

**代码架构**:

| 特性 | Claude Code | Codex |
|------|-------------|-------|
| **风格** | 扁平函数式 | 面向对象管道类 |
| **配置** | 各模块独立常量 | 集中式配置 |
| **数据结构** | 简单字典 | 数据类 (dataclasses) |
| **CLI** | 简单脚本 | argparse 完整 CLI |
| **验证** | 无输入验证 | 模型一致性验证 |
| **工程质量** | 简洁实用 | 更工程化、可配置 |

### 测试行为差异

| 方面 | Claude Code | Codex |
|------|-------------|-------|
| **端到端测试** | 主动测试整个脚本，确保管道可用 | 完成实现但未测试或运行 |
| **环境设置** | 自动处理依赖和环境问题 | 指示用户手动 pip install 和运行 |
| **错误处理** | 主动修复遇到的问题 | 遇到错误后修复 |
| **首 token 延迟** | 较短 | 可能高达一分钟 |
| **交互风格** | 更少解释，直接执行 | 更 verbose，详细解释计划 |
| **任务完成时间** | 更快 | 较慢 |

### 实验结果

使用 GPT-5.4 作为 LLM-as-a-judge，在四个标准上比较两款管道的答案：

- **Correctness (正确性)**: 答案是否准确
- **Completeness (完整性)**: 答案是否涵盖问题的所有方面
- **Relevance (相关性)**: 答案是否与问题相关
- **Conciseness (简洁性)**: 答案是否简洁明了

**结果统计 (100 个问题)**:

| 结果 | 数量 | 比例 |
|------|------|------|
| Claude Code 胜出 | 42 | 42% |
| Codex 胜出 | 33 | 33% |
| 平局 | 25 | 25% |

**胜出原因分析**:

Claude Code 主要胜在其**更宽松的置信度门控**和可能略高的生成温度 (0.2 vs Codex 管道的 0.1)。这意味着 Claude Code 更愿意尝试回答边缘问题，而 Codex 更倾向于保守地返回"信息不足"。

### 实验局限性与注意事项

这是一个非常简单的实验设置，主要目的是观察两款编码助手在实施相同封闭任务时的不同方法。在专业设置中：

- **开发者决策**: 整体架构的选择（分块方法、向量数据库、检索策略等）应该由开发者决定
- **迭代改进**: 开发此类系统需要更多测试和迭代改进，使用更可靠的测试集和验证
- **初级开发者现实**: 对于不太熟悉构建 RAG 管道的初级开发者来说，将这些决策交给 AI 是很自然的

## 10. 选择建议: 如何做出决定

### 核心观点: 任选其一都不会错

> 我认为选择 Claude Code 还是 Codex 都不会是致命错误的决定。两款工具都提供了与现有产品相比强大的模型，并且在完成任务方面达到相似的程度。

两个影响作者决策的主要因素：

1. **Anthropic 生态系统吸引力**
2. **$100/月的中档定价**

### 生态系统对比

| 方面 | Anthropic (Claude) | OpenAI |
|------|-------------------|--------|
| **整体感** | 完整、有凝聚力的生态系统 | 分散的、碎片化的产品 |
| **主要产品** | Claude Chat + Claude Code + Claude Cowork | Codex + ChatGPT + 其他 |
| **创新方向** | 渐进式、安全导向的 AI Agent 功能 | 激进、快速迭代的新模型 |
| **用户体验** | 精心打磨、一致 | 多变、不一致 |
| **生态系统吸引力** | **高** - 类似 Apple 的体验 | **低** - 缺乏整体感 |

### 定价对比

| 级别 | Claude Code | Codex |
|------|-------------|-------|
| **入门级** | $20/月 | $20/月 |
| **中级** | **$100/月 (Max 5x)** | **无** |
| **高级** | $200/月 | $200/月 |

**关键洞察**: Claude Code 的中档 $100/月计划是显著优势。 Codex 直接从 $20 跳到 $200，而 Claude Code 提供了中间选项，相信 Max 5x 计划 ($100/月) 对大多数开发者来说已经足够。

### 技能与插件生态系统

| 方面 | Claude Code | Codex |
|------|-------------|-------|
| **Skills 兼容性** | ✅ 原生支持 | ✅ 兼容 |
| **Skills 中心命名** | 大多以 Claude 命名 | 较少 |
| **Plugins 支持** | 早期支持 | 近期支持 |
| **Plugin 兼容性** | 良好 | 较低 |
| **可用 Plugins 数量** | 更多 | 较少 |

**关键洞察**: 虽然 Skills 在两者之间兼容，但大多数技能中心和仓库都以 Claude Code 命名。这反映了 Claude Code 的社区采用率和人气。

### 实际选择建议

#### 选择 Claude Code 如果你:

- ✅ 重视 Anthropic 的生态系统整合 (Claude Chat + Cowork + Code)
- ✅ 需要 $100/月的中档定价选项
- ✅ 喜欢"资深开发者伙伴"的交互风格
- ✅ 需要处理长期、复杂的任务 (12 小时时间跨度)
- ✅ 偏好更活跃的社区和技能资源

#### 选择 Codex 如果你:

- ✅ 你已经是 OpenAI 生态系统的重度用户
- ✅ 重视首次尝试的准确性高于交互体验
- ✅ 喜欢"承包商"风格的直接、高效交互
- ✅ 任务主要是 Web 开发和全栈项目
- ✅ 对 5 小时 50 分钟的任务时间跨度足够满意

#### 最实用的测试方法

> 如果你不确定选择哪款，我建议在你的相关编程领域试用两款产品的 $20/月版本，最好是在几个可验证的任务上进行测试。

### 最后的提醒

> 最后，请记住，与 AI 相关的任何事物一样，格局每几个月就会发生剧烈变化。虽然你现在可能喜欢其中一款，但三个月后，agent 的行为可能漂移，或者新模型可能上市。

> AI 领域很少有确定的全球性答案，而这个话题也不是其中之一 ;)
