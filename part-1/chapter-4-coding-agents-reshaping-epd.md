# 第4章：Coding Agents 如何重塑工程、产品与设计（EPD）

## 导言

在软件公司中，EPD（Engineering、Product、Design）的共同目标始终只有一个：**交付能解决业务问题、并被用户真正使用的软件**。无论角色如何划分，最终产物都是代码。

当 Coding Agents 让“写代码”本身变得前所未有地便宜时，软件团队的流程和角色分工就会发生结构性变化。本章将围绕两个问题展开：

1. 构建流程的瓶颈从哪里迁移到了哪里？
2. 工程、产品、设计三类角色分别会被如何重塑？

---

## 一、流程之变：从“先文档后实现”到“先原型后评审”

### 1.1 传统意义上的 PRD 流程正在失效

在“Agent 时代”之前，常见流程是：

1. 产品提出想法
2. 产品撰写 PRD
3. 设计基于 PRD 出稿
4. 工程将设计实现为代码

这套流程成立的前提，是“实现成本高、跨职能协作成本也高”。PRD 的核心作用是跨角色对齐。

但现在，Coding Agents 可以把想法快速转成可运行原型，传统的“PRD → 设计稿 → 代码”的线性流水线不再是默认最优路径。

> 所谓“PRD 已死”，并不是需求表达不重要，而是“以 PRD 作为唯一起点的瀑布式交付”正在失去主导地位。

### 1.2 瓶颈从实现转向评审

“任何人都能写代码”的真实含义是：

- 原型数量会显著上升
- 低质量方案也会显著上升
- 真正稀缺的能力从“能做出来”转向“能判断好坏”

因此，EPD 的核心价值开始变成评审与裁决：

- **工程评审**：架构是否可扩展、性能是否可接受、系统是否稳健
- **产品评审**：是否真的解决用户痛点，是否匹配业务目标
- **设计评审**：交互是否直觉、信息层级是否清晰、体验是否一致

当原型生成几乎零门槛时，团队的主要限制因素就是“高质量评审能力的供给”。

### 1.3 PRD 没有消失，而是“形态升级”

传统 PRD 流程在弱化，但“需求意图文档”反而更重要。

原型提交评审时，如果没有意图说明，评审者很难判断：

- 某处行为是“有意设计”还是“生成偶然”
- 某个取舍是“业务约束”还是“实现妥协”

所以，未来更合理的交付单元是：

- **原型（可运行）** + **意图说明（可审阅）**

意图说明可以是文档，也可能是结构化、可版本化的 Prompt 记录。PRD 没死，只是从“前置大文档”演化为“伴随式说明层”。

## 二、角色之变：从职能边界到能力组合

### 2.1 通才价值显著上升

具备产品、工程、设计综合判断力的人，会在 Agent 时代获得更高杠杆。

原因很直接：沟通是最大的隐性成本。一个能串起三种视角的人，可以减少反复对齐，并直接驱动 Agent 快速迭代，从而把“个人产能上限”推高。

### 2.2 使用 Coding Agents 将从“加分项”变成“任职要求”

- PM 若会用 Agent，可直接做可交互原型验证假设
- 设计师若会用 Agent，可在代码里迭代而非只停留在静态稿
- 工程师若会用 Agent，可把精力从样板实现转向系统思考

当采用成本低、收益高时，不采用就会成为明显劣势。

### 2.3 好 PM 的杠杆更大，差 PM 的破坏性也更大

Agent 让“把想法做出来”变容易，也放大了想法质量差异：

- 好 PM：更快验证高价值方向
- 差 PM：更快制造低价值原型并占用团队评审资源

更现实的问题是，低质量原型容易产生“沉没成本推动”——“都做出来了，不如上线吧”。这会导致产品膨胀与体验劣化。

### 2.4 系统思维成为核心分水岭

执行成本下降后，真正稀缺的是系统思维。

- **工程**：对服务、API、数据库、边界与演进路径有整体模型
- **产品**：能区分用户“表达的诉求”与“真实的需求”
- **设计**：能解释某种交互为何可学、可用、可持续

系统思维既决定“前置决策质量”，也决定“后置评审质量”。两端价值都在上升。

### 2.5 人人都需要产品感（Product Sense）

Agent 只能执行被表达的目标。目标错了，产出越快，返工越大。

因此，工程、设计、产品都需要基本产品感，至少要能判断：

- 这个功能解决的核心问题是什么
- 成功指标是什么
- 边界条件与失败路径是什么

产品感越强，所需说明越短、跨职能协作越快、评审效率越高。

### 2.6 专业化没有消失，但门槛抬高了

未来仍然需要顶级专家，例如：

- 深度系统架构师
- 对用户与市场有深刻洞察的资深 PM
- 能构建复杂交互范式的高级设计师

但纯专业化角色要持续创造高价值，必须同时满足：

1. 领域能力足够深
2. 评审速度足够快
3. 跨团队沟通足够强

### 2.7 组织中将出现两类核心角色：Builder 与 Reviewer

**Builder（构建者）**
- 有基本产品判断
- 会熟练使用 Coding Agents
- 有足够的设计与工程基础
- 能在护栏（测试、组件库、规范）下将小功能快速推到生产

**Reviewer（评审者）**
- 在工程/产品/设计某一侧具有高阶系统思维
- 能快速识别风险、做架构级与体验级判断
- 能在大量原型涌入时给出高质量筛选与方向矫正

很多岗位会在这两类角色之间流动，而不是被传统头衔严格切开。

---

## 三、一个关键结论：每个角色都在被增强

很多人会觉得“我的岗位最受益于 Coding Agents”，这种判断往往都成立。

因为真正被放大的不是某个单点技能，而是“理解产品 + 驱动实现 + 组织评审”的闭环能力。这个闭环可以从工程、产品、设计任一背景出发形成。

差别不在于你来自哪个岗位，而在于你是否具备以下组合：

- 对产品与用户有真实洞察
- 对技术与实现边界有现实判断
- 能把想法快速外化为可评审原型
- 能在高频反馈中持续迭代并收敛

---

## 本章小结

Coding Agents 正在让 EPD 进入一个“实现便宜、判断昂贵”的新阶段：

- 传统 PRD 驱动的线性流程在退场
- 原型驱动 + 文档伴随的评审流程在上升
- 通才、系统思维、产品感成为组织关键资产
- 团队角色从“按职能分工”走向“Builder/Reviewer 协同”

这不是工程、产品、设计谁替代谁的问题，而是三者在同一条价值链上被重新编排。
