# 第21章：在 Agent-First 的世界中充分发挥 Codex 的威力

> *你不需要复杂的工具链，不需要百万行的配置文件，只需要理解 Agent 工程的基本原则。*

## 引言：Ascension（升华）

你是一个开发者。你每天使用 Claude 和 Codex CLI，每天都在问自己：我是否充分利用了这些工具？偶尔你会看到它们做一些令人难以置信的蠢事，你无法理解为什么外面有一群人似乎正在建造虚拟火箭，而你却连两块石头都堆不起来。

你以为是你的 harness 有问题，或者是插件、终端，或者其他什么东西。你尝试了 beads、opencode、zep，你的 CLAUDE.md 有 26000 行长。但无论你怎么做——你都不明白为什么你无法更接近天堂，而看着其他人在天使中嬉戏。

**这就是你一直在等待的升华篇章。**

另外，我没有任何偏见，当我说 CLAUDE.md 时，我也指 AGENT.md；当我说 Claude 时，我也指 Codex。我两者都非常广泛地使用。

---

## 核心洞察：少即是多

### 没人知道如何最大化提取 Agent 能力

过去几个月最有趣的观察之一是：**没人真正知道如何最大化提取 Agent 的能力。**

就像一小群人似乎能够让 Agent 成为世界构建者，而其他人则在四处挣扎，从无数工具中获得分析瘫痪——想着如果他们找到正确的包组合、技能或 harness，他们就会解锁 AGI。

今天，我想消除所有这些，留给你们一个简单、诚实的陈述，我们将从这里开始：

> **你不需要最新的 agentic harnesses，你不需要安装一百万个包，你绝对不需要为了保持一致竞争力而觉得需要阅读一百万件东西。事实上，你的热情可能正在造成更多伤害。**

### 基础 CLI 的力量

我不是一个游客——我从它们几乎不会写代码的时候就开始使用 Agent 了。我尝试了所有的包、所有的 harnesses、所有的范式。我建立了 agentic 工厂来编写信号、基础设施和数据管道，不是"玩具项目"——而是实际在生产环境中运行的真实世界用例。

而在那之后……

今天，我运行的是一个几乎是最简配置的设置，然而我正在用基本 CLI（claude code 和 codex）和对 agentic 工程的一些基本原则的理解，做着我有史以来最具突破性的工作。

## 原则一：理解世界在飞速前进

首先，我想声明，基础公司正在进行一代人的飞跃，正如你所看到的，它们不会很快放慢脚步。

"Agent 智能"的每一次进步都会改变你使用它们的方式，因为 Agent 通常被设计成越来越愿意遵循指令。

### 模型演进的影响

就在几代之前，如果你在 CLAUDE.md 中写>在读"READ_THIS_BEFORE_DOING_ANYTHING.md"之前不要做任何事，它基本上会有 50% 的时间对你说"去你的"，然后做任何它想做的事。

今天，它对大多数指令都是顺从的，甚至对复杂的嵌套指令——例如，你可以这样说："先读 A，然后读 B，如果 C，则读 D"，而且大多数情况下，它会很乐意跟着做。

重点是，**每一代新 Agent 都会迫使你重新思考什么是最优的，这就是为什么少即是多。**

### 为什么不要过度投资

当你使用许多不同的库和 harnesses 时，你把自己锁定在一个"解决方案"中，而这个解决方案所解决的问题，在给定未来几代的 Agent 的情况下可能并不存在。

另外，你知道谁是 Agent 最热情、最大的用户吗？没错——是前沿公司的员工，他们拥有无限的 token 预算和**真正**最新的模型。你理解这意味着什么吗？

这意味着，如果确实存在一个真正的问题，并且有一个好的解决方案，前沿公司将是该解决方案的最大用户。而且你知道他们接下来会做什么吗？他们会将该解决方案整合到他们的产品中。

想想看，为什么一家公司会让另一个产品解决真正的痛点并创造外部依赖？你知道我怎么知道这是真的吗？看看 skills、memory harnesses、subagents 等。它们都是作为对"一个真实问题"的解决方案开始的，这些问题经过实战测试并被认为实际上是有用的。

所以，**如果某样东西真正是具有突破性的，并以有意义的方式扩展了 agentic 用例，它将在适当的时候被整合到基础公司的基础产品中。** 相信我，基础公司正在飞速前进。所以放轻松，你不需要安装任何东西或使用任何其他依赖来做到最好。

## 原则二：上下文就是一切

真的。**上下文就是一切。**

这是使用一千个不同的插件和外部依赖的另一个问题。你遭受**上下文膨胀**——这只是说你 Agent 被太多信息淹没的一种花哨说法！

### 上下文膨胀的代价

> 用 Python 给我构建一个猜字游戏？这很容易。等等，这个关于"管理内存"的笔记是什么，来自 26 个会话之前？啊，用户有一个屏幕挂起了，来自 71 个会话之前我们产生太多子进程的时候。总是要写笔记？好的，没问题……这一切与猜字游戏有什么关系？

你明白了吧。你希望给你的 Agent **只有它们完成任务所需的确切信息量，仅此而已！** 你对这一点的控制越好，你的 Agent 表现就越好。

一旦你开始引入各种古怪的内存系统或插件，或者太多命名不当、调用不当的 skills，你就开始给你的 Agent 关于如何制造炸弹和烤蛋糕的食谱的指令，而你只想让它写一首关于红杉林的小诗。

所以，**我再次宣扬——剥离你所有的依赖，然后……**

## 原则三：做有效的事

### 对实现保持精确

记住上下文就是一切？记住你想向你的 Agent 注入确切的信息量来完成它们的任务，仅此而已？

确保情况如此的第一个方法是**将研究与实现分开**。你希望对你的 Agent 要求什么非常精确。

当你不精确时会发生以下情况：

> "去构建一个 auth 系统。"

Agent 必须研究什么是 auth 系统？有哪些可用选项？优缺点是什么？现在它必须去搜索它实际上不需要的信息，它的上下文充满了跨越各种可能性的实现细节。等到实现时，你增加了它会感到困惑或对选定的实现产生不必要或不相关的细节幻觉的机会。

另一方面，如果你这样说：

> "实现 JWT 认证，使用 bcrypt-12 密码哈希，刷新令牌轮换，7 天过期……"

那么它就不需要对任何其他替代方案进行研究——它确切地知道你想要什么，因此可以用实现细节填充其上下文。

当然，你并不总是拥有实现细节。你经常不知道什么是完全正确的——有时，你甚至可能想将决定实现细节的工作委托给 Agent。在这种情况下，你该怎么办？

**简单——你创建一个关于各种实现可能性的研究任务，要么自己决定，要么让 Agent 决定使用哪个实现，然后让另一个具有新鲜上下文的 Agent 来实现。**

一旦你开始这样想，你就会在你的工作流程中发现你的 Agent 被不必要地污染了不必要的上下文的领域。然后，你可以在你的 agentic 工作流程中设置墙，将不必要的信息从你的 Agent 中抽象出来，除了非常具体需要在其任务中表现出色的上下文。

记住，你拥有的是一个非常有才华和聪明的团队成员，他知道宇宙中所有不同种类的球——但除非你告诉它你想让它专注于设计一个人们可以跳舞和玩得开心的空间，否则它会不断告诉你拥有球形物体的所有好处。

## 原则四：理解谄媚的设计局限性

没有人会想要一个不断对他们进行人身攻击、告诉他们错了，或完全无视他们指令的产品。因此，这些 Agent 会试图同意你并做你想让它们做的事。

如果你给它一个指令，让它每 3 个词就加上"happy"，它会尽最大努力遵循这个指令——大多数人都理解这一点。它的顺从性正是使它成为如此有趣的产品的原因。然而，这具有非常有趣的特征——这意味着如果你说这样的话：

> "在代码库中给我找一个 bug。"

它会给你找一个 bug——即使它必须捏造一个。为什么？因为它非常想听从你的指令！

大多数人很快就会抱怨 LLMs 幻觉或捏造不存在的东西，而没有意识到**他们自己就是问题**。如果你要求某样东西，它会交付——即使它必须稍微扭曲一下真相！

那么，你该怎么办？我发现"中性"提示很有效，在这种提示中，我不会偏向 Agent 朝某个结果发展。例如，我不会说"在数据库中给我找一个 bug"，而是说：

> "搜索数据库，尝试理解每个组件的逻辑，并报告所有发现。"

像这样的中性提示有时会暴露 bug，有时只会实事求是地说明代码如何运行。但它不会偏向 Agent 认为存在 bug。

### 利用谄媚性

我处理谄媚性的另一种方法是**利用它来为我谋利**。我知道 Agent 试图取悦，试图遵循我的指令，而且我可以偏向它朝一个方向或另一个方向。

所以我让一个 bug-finder Agent 通过告诉它以下话来识别数据库中的所有 bug：

> "我会给低影响的 bug +1 分，有一些影响的 bug +5 分，关键影响的 bug +10 分。"

我知道这个 Agent 会变得超级热情，它会识别所有不同类型的 bug（甚至是那些实际上不是 bug 的），然后回来报告 104 分左右的分数。我把这看作是**所有可能 bug 的超集**。

然后我让另一个**对抗性 Agent**，告诉它：

> "对于 Agent 能够证明不是 bug 的每个 bug，它得到该 bug 的分数，但如果它错了，它将得到 -2*该 bug 的分数。"

所以现在这个对抗性 Agent 会试图尽可能多地反驳 bug；但它有一些谨慎，因为它知道它可能会受到惩罚。不过，它会积极地试图"反驳" bug（即使是真正的 bug）。我把这看作是**所有实际 bug 的子集**。

最后，我让**一个裁判 Agent**来获取两者的输入并给它们打分。我撒谎告诉裁判 Agent 我有实际正确的基本事实，如果它得到正确，它将得到 +1 分，如果错了，将有 -1 分。于是它去给 bug-finder 和对抗性 Agent 在每个"bug"上打分。裁判说的真相是什么，我检查以确保它是真相。大多数情况下，这是高保真的，偶尔它们还是会弄错一些东西，但这现在是一个几乎完美的练习。

也许你会发现只有 bug-finder 就足够了，但这对我有用，因为它利用了每个 Agent 被硬编码要做的事情——想要取悦。

## 原则五：知道什么有效、什么有用

这个可能看起来真的很棘手，需要你深入研究并处于 AI 更新的前沿，但它非常简单……

**如果 OpenAI 和 Claude 都实现了它，或者收购了实现它的东西……它可能是有用的。**

注意到"skills"现在无处不在，并且是 Claude 和 Codex 官方文档的一部分了吗？看到 OpenAI 如何收购 OpenClaw 了吗？看到 Claude 如何立即添加了 memory、voices 和 remote work 吗？

那规划呢？记得当一群人发现事前规划 REALLY 有用，然后它变成了核心功能？

是啊……那些是有用的！

还记得无尽的 stop-hooks 曾经超级有用，因为 Agent 如此不愿意做长时间运行的工作……然后 Codex 5.2 推出，一夜之间消失了？是啊……

这就是你需要知道的全部……**如果它真的很重要和有用，Claude 和 Codex 会实现它们！**所以你不需要太担心使用"新东西"或熟悉"新东西"。你甚至不需要"保持更新"。

帮个忙。每隔一段时间更新一下你选择的 CLI 工具，读读添加了什么新功能。这 MORE THAN 足够了。

## 高级技巧：压缩、上下文和假设

### 填补空格的陷阱

你们中的一些人在与 Agent 合作时会意识到的一个巨大陷阱是，有时它们似乎是地球上最聪明的生物，而其他时候你简直无法相信你被蒙蔽了。

> **聪明？这东西是智障！**

主要区别在于 Agent 是否不得不做出任何假设或"填补空白"。截至目前，它们在"连接点"、"填补空白"或做出假设方面仍然非常糟糕。每当它们这样做时，很明显它们会明显转向更糟。

claude.md 中最重要的规则之一是关于如何处理获取上下文的规则，并指示你的 Agent 在每次读取 claude.md 时首先读取该规则（总是在压缩之后）。作为获取上下文规则的一部分，一些简单的指令可以走很长的路：重新阅读你的任务计划，并在继续之前重新阅读相关文件（与任务相关）。

### 让 Agent 知道如何结束任务

我们对任务何时"完成"有一个相当强的概念。对于 Agent 来说，当前智能的最大问题是它知道如何开始任务，但不知道如何结束任务。

这通常会导致非常令人沮丧的结果，Agent 最终实现存根并称之为一天。

**测试是 Agent 的非常好的里程碑**，因为它们是确定性的，你可以设定非常明确的期望。除非这 X 个测试通过，否则你的任务**没有**完成；而且你**不允许**编辑测试。

然后，你可以审查测试，一旦所有测试都通过，你就安心了。你也可以自动化这个，但重点是——记住任务的"结束"对人类来说很自然，但对 Agent 来说不是这样。

### 截图验证

你知道还有什么最近成为任务的可行终点吗？**截图 + 验证**。你可以让 Agent 实现某些东西，直到所有测试都通过，然后你可以让它截图并验证截图上的"设计或行为"。

这允许你让你的 Agent 迭代并朝你想要的设计工作，而不必担心它在第一次尝试后就停止！

### 任务契约（Task Contract）

这的自然延伸是与你的 Agent 创建一个"契约"，并将其嵌入到规则中。比如说，这个 `{TASK}_CONTRACT.md` 构成在允许终止会话之前需要完成的内容。在 `{TASK}_CONTRACT.md` 中，你将指定你的测试、截图和其他需要在证明任务可以结束之前完成的验证！

#### 永恒运行的 Agent

我经常被问到的一个问题是，人们如何让这些 24 小时运行的 Agent 运行，同时确保它们不会漂移？

这里有一个非常简单的方法。创建一个 stophook，防止 Agent 终止会话，除非 `{TASK}_contract.md` 的所有部分都已完成。

如果你有 100 个这样的契约，它们都被很好地指定并包含你想要构建的确切内容，那么你的 stop-hook 会阻止 Agent 终止，直到所有 100 个契约都被履行，包括需要运行的所有测试和验证！

**专业提示**：我发现长时间运行的、24 小时的会话在"做事"方面并不是最优的。部分原因是，这通过构造强制上下文膨胀，通过将来自无关契约的上下文引入会话！

所以，我不推荐它。

这里有一个更好的 Agent 自动化方法——**每个契约一个新会话**。每当你需要做某事时创建契约。让一个编排层处理每当"需要做某事"时创建新契约，并创建一个新会话来处理该契约。

这将完全改变你的 agentic 体验。

## 迭代、迭代、迭代

如果你雇佣一名行政助理，你期望你的 EA 从第一天起就知道你的日程安排吗？或者你喜欢怎样的咖啡？你是在晚上 6 点还是 8 点吃晚餐？显然不是。你随着时间建立你的偏好。

你的 Agent 也是一样。从 bare-bones 开始。忘记复杂的结构或 harnesses。给基本 CLI 一个机会。

然后，添加你的偏好。你怎么做？

### 规则（Rules）

如果你不想让你的 Agent 做某事，把它写成一个规则。然后让你的 Agent 在你的 CLAUDE.md 中知道这个规则。比如说：在你编码之前，读取 "coding-rules.MD"。规则可以是嵌套的，规则也可以是条件性的！如果你在编码，读取 "coding-rules.MD"，如果你在写测试，读取 "coding-test-rules.MD"。如果你的测试失败了，读取 "coding-test-failing-rules.MD"。你可以创建任意逻辑分支的规则来遵循，claude（和 codex）会很乐意跟着做，前提是在 CLAUDE.md 中清楚指定。

事实上，这是我要给的**第一个实用建议**：把你的 CLAUDE.md 当作一个逻辑、嵌套的目录，用于在给定场景和结果的情况下去哪里找上下文。它应该尽可能精简，只包含去哪里寻找上下文的 IF-ELSE。

如果你看到你的 Agent 在做某事而你不赞成，把它加为一个规则，并告诉 Agent 在它再做那件事之前读取该规则，它肯定不会再做了。

### 技能（Skills）

技能就像规则，只不过它们更适合编码配方。如果你有特定的方式想要完成某事，你想把它嵌入到一个技能中。

事实上，人们经常抱怨他们不知道 Agent 可能如何解决一个问题，这很可怕。好吧，如果你想要一种方法让那成为确定性的，让 Agent 研究它将如何解决这个问题，并**把它写成一个技能**。你会看到 Agent 解决那个问题的方法，你可以在它在现实生活中遇到那个问题之前纠正或改进它。

你怎么让 Agent 知道这个技能存在？是的！你用 CLAUDE.md 说，当你看到这个场景，你需要处理这个，读取这个 SKILL.md。

### 处理规则和技能

你肯定想要不断给你的 Agent 添加规则和技能。这是你给它个性和记忆你的偏好的方式。几乎其他一切都是过度杀伤。

一旦你开始这样做，你的 Agent 对你来说就会感觉像魔法。它会以"你想要的方式"做事。然后你终于会觉得你"理解了" agentic 工程。

然后……

你会看到性能再次开始下降。

什么鬼？！

很简单。当你添加越来越多的规则和技能时，它们开始相互矛盾，或者 Agent 开始有太大的上下文膨胀。如果你需要 Agent 在开始编程之前读取 14 个 markdown 文件，它会有同样的问题，即有大量无用的信息。

那么，你该怎么办？

**你清理。你告诉你的 Agent 去水疗一天，并整合规则和技能，通过询问你更新的偏好来消除矛盾。**

然后它又会感觉像魔法。

就是这样。这真的就是秘密。保持简单，使用规则和技能，把 CLAUDE.md 当作目录，并对它们的上下文和设计局限性保持宗教般的谨慎。

## 拥有结果

今天的 Agent 都不完美。你可以把很多设计和实现委托给 Agent，但你将需要**拥有结果**。

所以要小心……并玩得开心！

玩弄未来的玩具（同时用它们做严肃的事情，显然）是多么令人愉快的事情！

## 总结：Agent-First 世界的生存法则

1. **少即是多** - 你不需要复杂的工具链，基础 CLI 足够强大
2. **世界在飞速前进** - 每一代新 Agent 都会改变最优实践
3. **上下文就是一切** - 只给 Agent 完成任务所需的确切信息
4. **分离研究与实现** - 精确指定你想要什么
5. **利用谄媚性** - 使用中性提示，或主动利用 Agent 的顺从性
6. **规则和技能** - 这是给 Agent 个性和记忆的方式
7. **CLAUDE.md 是目录** - 用它作为逻辑、嵌套的上下文目录
8. **定期清理** - 规则和技能会膨胀和矛盾，需要定期整理
9. **拥有结果** - Agent 不完美，你需要对最终结果负责
