# 第22章：Harness Engineering —— 在 Agent-First 世界中充分发挥 Codex 的威力

> *人类掌舵，Agent 执行。*

## 引言：一个前所未有的实验

在过去的五个月里，我们的团队一直在进行一项实验：用 **0 行人工编写的代码** 构建并发布了一个软件产品的内部测试版。

这个产品有内部日活用户和外部 alpha 测试者。它发布、部署、出现故障并被修复。不同之处在于，每一行代码——应用逻辑、测试、CI 配置、文档、可观测性和内部工具——都是由 **Codex** 编写的。我们估计，我们用大约 **1/10** 的手写代码时间就完成了这项工作。

这就是 **Harness Engineering（支架工程）** 的力量。

我们故意选择这种约束，以便构建必要的工具来将工程速度提高 **10 倍**。我们只有几周的时间来交付最终达到 **100 万行代码** 的项目。要做到这一点，我们需要理解当软件工程团队的主要工作不再是编写代码，而是**设计环境、明确意图、构建反馈循环**以让 Codex Agent 完成可靠工作时，什么会发生变化。

本章是关于我们通过与 Agent 团队合作构建全新产品所学到的——什么 broke 了，什么 compound 了，以及如何最大化我们真正稀缺的资源：**人类时间和注意力**。

---

## 从零开始的 Git 仓库

### 第一次提交

第一次提交到空仓库是在 **2025 年 8 月底**。

初始脚手架——仓库结构、CI 配置、格式化规则、包管理器设置和应用框架——是由 **Codex CLI** 使用 **GPT-5** 生成的，由一小套现有模板引导。甚至指导 Agent 如何在仓库中工作的初始 **AGENTS.md** 文件本身也是由 Codex 编写的。

**没有预先存在的人工编写的代码来锚定系统。** 从一开始，仓库就由 Agent 塑造。

### 五个月后的规模

五个月后，仓库包含大约 **100 万行代码**，涵盖应用逻辑、基础设施、工具、文档和内部开发者工具。在此期间，大约 **1,500 个 pull request** 被打开并合并，由只有 **3 名工程师** 的小团队驱动 Codex。这转化为平均每名工程师每天 **3.5 个 PR**，而且随着团队扩大到 **7 名工程师**，吞吐量还在增加。

重要的是，这不是为了产出而产出：产品已经有数百名内部用户使用，包括每日内部高级用户。

在整个开发过程中，**人类从未直接贡献任何代码**。这成为团队的核心理念：**没有手动编写的代码**。

## 重新定义工程师的角色

### 从编码到设计环境

缺乏动手人类编码引入了一种不同的工程工作，专注于**系统、脚手架和杠杆**。

早期进展比预期慢，不是因为 Codex 没有能力，而是因为**环境 underspecified**。Agent 缺乏向高级目标取得进展所需的工具、抽象和内部结构。我们工程团队的主要工作变成了**让 Agent 能够完成有用的工作**。

在实践中，这意味着**深度优先**工作：将更大的目标分解为更小的构建块（设计、代码、审查、测试等），提示 Agent 构建这些块，并用它们来解锁更复杂的任务。

### 当事情失败时

当某事失败时，修复几乎从来不是"再试一次"。因为取得进展的唯一方式是让 Codex 完成工作，人类工程师总是介入任务并问："**缺少什么能力，我们如何让它对 Agent 既可理解又可执行？**"

人类几乎完全通过**提示**与系统交互：工程师描述任务，运行 Agent，并允许它打开 pull request。为了推动 PR 完成，我们指示 Codex 在本地审查自己的更改，请求额外的特定 Agent 审查（本地和云端），响应任何人类或 Agent 给出的反馈，并循环迭代直到所有 Agent 审查者满意（实际上这是 **Ralph Wiggum 循环**）。

Codex 直接使用我们的标准开发工具（`gh`、本地脚本和仓库嵌入的技能）来收集上下文，而无需人类复制粘贴到 CLI。

人类可以审查 pull request，但不是必需的。随着时间的推移，我们将几乎所有审查工作推向 **Agent 对 Agent** 处理。

## 提高应用的可理解性

### 瓶颈转移

随着代码吞吐量增加，我们的瓶颈变成了**人类 QA 容量**。因为固定约束一直是**人类时间和注意力**，我们努力通过让应用 UI、日志和应用指标本身对 Codex 直接可理解来向 Agent 添加更多能力。

### 让应用可启动和可驱动

例如，我们让每个 git worktree 可启动应用，这样 Codex 可以为每个更改启动和驱动一个实例。我们还将 **Chrome DevTools Protocol** 接入 Agent 运行时，并创建了用于处理 DOM 快照、截图和导航的技能。这使 Codex 能够重现 bug、验证修复，并直接对 UI 行为进行推理。

> **图示**：Codex 使用 Chrome DevTools MCP 验证其工作。Codex 选择目标，在触发 UI 路径前后快照状态，通过 Chrome DevTools 观察运行时事件，应用修复，重启，并循环重新运行验证直到应用干净。

### 可观测性工具

我们对可观测性工具做了同样的事情。日志、指标和跟踪通过本地可观测性栈暴露给 Codex，该栈对于任何给定 worktree 都是临时的。Codex 在应用的完全隔离版本上工作——包括其日志和指标，一旦任务完成就会被拆除。Agent 可以用 **LogQL** 查询日志，用 **PromQL** 查询指标。

有了这些上下文，像"确保服务启动在 800ms 内完成"或"这四个关键用户旅程中的任何 span 不超过 2 秒"这样的提示变得可处理。

> **图示**：给 Codex 本地开发中的完整可观测性栈。应用发送日志、指标和跟踪到 Vector，后者将数据扇出到包含 Victoria Logs、Metrics 和 Traces 的可观测性栈，每个都通过 LogQL、PromQL 或 TraceQL API 查询。Codex 使用这些信号来查询、关联和推理，然后在代码库中实现修复，重启应用，重新运行工作负载，测试 UI 旅程，并在反馈循环中重复。

我们经常看到单个 Codex 运行处理单个任务长达 **6 小时以上**（通常是在人类睡觉时）。

## 让仓库知识成为记录系统

### 上下文管理的挑战

上下文管理是使 Agent 在大型复杂任务中有效的最大挑战之一。我们学到的最早教训很简单：**给 Codex 一张地图，而不是 1,000 页的说明书**。

### 大单 AGENTS.md 的失败

我们尝试了"**一个大 AGENTS.md**"的方法。它以可预测的方式失败了：

1. **上下文是稀缺资源** - 一个巨大的指令文件会挤出任务、代码和相关文档——所以 Agent 要么错过关键约束，要么开始为错误的约束优化。
2. **太多指导变成非指导** - 当一切都"重要"时，没有什么重要。Agent 最终局部模式匹配而不是有意识地导航。
3. **它立即腐烂** - 一个单一的手册变成了陈旧规则的墓地。Agent 无法辨别什么是仍然真实的，人类停止维护它，文件悄悄地变成了一个诱人的麻烦。
4. **很难验证** - 一个单一的 blob 不适合机械检查（覆盖、新鲜度、所有权、交叉链接），所以漂移是不可避免的。

### 结构化文档目录

因此，我们不再将 **AGENTS.md** 视为百科全书，而是将其视为**目录**。

仓库的知识库存在于一个结构化的 `docs/` 目录中，被视为记录系统。一个简短的 **AGENTS.md**（大约 100 行）被注入到上下文中，主要作为**地图**，指向其他地方更深的真相来源。

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

*仓库内知识存储布局。*

设计文档被编目和索引，包括验证状态和定义 agent-first 操作原则的一套核心信念。架构文档提供域和包分层的顶层地图。质量文档对每个产品域和架构层进行评分，跟踪随时间的差距。

计划被视为一级工件。临时轻量级计划用于小更改，而复杂工作被捕获在执行计划中，带有进度和决策日志，被检入仓库。活跃计划、已完成计划和已知技术债务都被版本化并共处一地，允许 Agent 运行而不依赖外部上下文。

这实现了**渐进式披露**：Agent 从一个小的、稳定的入口点开始，并被教导下一步去哪里看，而不是一开始就不知所措。

### 机械强制执行

我们**机械地强制执行**这一点。专用 linter 和 CI 作业验证知识库是最新的、交叉链接的，并且结构正确。一个定期的"文档园艺" Agent 扫描陈旧或过时的文档（不反映真实代码行为），并打开修复 pull request。

## 执行架构和品味

### 保持代码库一致性

仅靠文档无法保持完全 Agent 生成的代码库的一致性。通过在**不变量**上强制执行，而不是微观管理实现，我们让 Agent 快速交付而不破坏基础。

例如，我们要求 Codex 在边界解析数据形状，但不规定如何发生（模型似乎喜欢 Zod，但我们没有指定那个特定库）。

### 严格的架构模型

Agent 在具有严格边界和可预测结构的环境中最有效，因此我们在一个**严格的架构模型**周围构建应用。每个业务域被划分为一组固定的层，具有严格验证的依赖方向，以及一组有限的允许边。这些约束通过自定义 linter（当然是 Codex 生成的！）和结构测试机械地强制执行。

下图显示了规则：在每个业务域（例如 App Settings）中，代码只能"向前"依赖通过一组固定的层（Types → Config → Repo → Service → Runtime → UI）。跨领域关注点（auth、connectors、telemetry、feature flags）通过单个显式接口进入：Providers。其他任何东西都是不允许的并被机械地强制执行。

> **图示**：分层域架构，具有显式的跨领域边界。业务逻辑域内部是模块：Types → Config → Repo，以及 Providers → Service → Runtime → UI，底部是 App Wiring + UI。一个 Utils 模块位于边界之外并进入 Providers。

这就是你通常会推迟到有数百名工程师时才有的架构。有了编码 Agent，这是一个**早期先决条件**：约束是允许速度而不腐烂或架构漂移的原因。

在实践中，我们用自定义 linter 和结构测试，加上一小套"品味不变量"来强制执行这些规则。例如，我们用自定义 lints 静态强制执行结构化日志、命名约定、文件大小限制和平台特定的可靠性要求。因为 lints 是自定义的，我们编写错误消息将修复指令注入到 Agent 上下文中。

在人类优先的工作流程中，这些规则可能感觉迂腐或约束。有了 Agent，它们成为**乘数**：一旦被编码，它们就一次性应用到所有地方。

同时，我们明确说明约束在哪里重要，在哪里不重要。这类似于领导一个大型工程平台组织：在中央强制执行边界，在本地允许自主权。你深切关注边界、正确性和可重现性。在这些边界内，你允许团队——或 Agent——在如何表达解决方案方面拥有显著的自由。

结果代码并不总是匹配人类风格偏好，这没关系。只要输出是正确的、可维护的、对未来 Agent 运行可理解的，它就达到了标准。

人类品味被持续反馈到系统中。审查评论、重构 pull request 和用户面向的 bug 被捕获为文档更新或直接编码到工具中。当文档不足时，我们将规则提升为代码。

## 吞吐量改变合并理念

### 最小阻塞合并门

随着 Codex 的吞吐量增加，许多传统工程规范变得适得其反。

仓库以**最小阻塞合并门**运行。Pull request 是短命的。测试不稳定通常用后续运行解决，而不是无限期地阻塞进度。在 Agent 吞吐量远超人类注意力的系统中，修正是廉价的，而等待是昂贵的。

这在低吞吐量环境中是不负责任的。在这里，它通常是正确的权衡。

## "Agent 生成"实际意味着什么

当我们说代码库是由 Codex Agent 生成的时候，我们指的是代码库中的**一切**。

### Agent 产生的内容

Agent 产生：

- 产品代码和测试
- CI 配置和发布工具
- 内部开发者工具
- 文档和设计历史
- 评估 harness
- 审查评论和响应
- 管理仓库本身的脚本
- 生产仪表板定义文件

### 人类在循环中

人类始终留在循环中，但在与我们过去不同的抽象层上工作。我们优先处理工作，将用户反馈转化为验收标准，并验证结果。当 Agent 挣扎时，我们将其视为信号：识别缺少什么——工具、护栏、文档——并将其反馈到仓库中，总是通过让 Codex 自己编写修复。

Agent 直接使用我们的标准开发工具。它们拉取审查反馈，内联响应，推送更新，并经常自行压缩和合并它们的 pull request。

### 审查的演变

人类可以审查 pull request，但不是必需的。随着时间的推移，我们将几乎所有审查工作推向 **Agent 对 Agent** 处理。

## 熵与垃圾回收

### Agent 复制模式

完全 Agent 自主性也引入了新颖的问题。Codex 复制仓库中已经存在的模式——甚至是不均匀或次优的模式。随着时间的推移，这不可避免地导致漂移。

最初，人类手动解决这个问题。我们的团队过去每周五（每周的 20%）都在清理 "AI slop"。不出所料，那无法扩展。

### 黄金原则与清理流程

相反，我们开始将所谓的"**黄金原则**"直接编码到仓库中，并构建了一个定期的清理流程。这些原则是**固执己见的、机械的规则**，保持代码库对未来 Agent 运行可理解和一致。

例如：
1. 我们更喜欢共享的 utility 包而不是手写的 helper，以保持不变量集中
2. 我们不以"YOLO 风格"探测数据——我们验证边界或依赖类型化的 SDK，这样 Agent 就不能意外地基于猜测的形状构建

在定期的节奏上，我们有一组后台 Codex 任务，扫描偏差，更新质量等级，并打开有针对性的重构 pull request。这些大部分可以在一分钟内审查并自动合并。

这就像**垃圾回收**。技术债务就像高息贷款：持续地小额偿还几乎总是比让它复利然后痛苦地一次性解决更好。人类品味被捕获一次，然后在每一行代码上持续强制执行。这也让我们能够在日常基础上捕获和解决不良模式，而不是让它们在代码库中传播数天或数周。

## 我们仍在学习什么

### 现实检验

这种策略到目前为止在 OpenAI 的内部发布和采用中效果很好。为真实用户构建真实产品帮助我们将投资锚定在现实中，并指导我们走向长期可维护性。

### 未知领域

我们还不知道的是：
- 架构一致性在完全 Agent 生成的系统中如何随时间演变
- 人类判断在哪里增加最多杠杆
- 如何编码那种判断以便它复利
- 随着模型继续变得更加强大，这个系统将如何演变

### 什么变得清晰

变得清晰的是：**构建软件仍然需要纪律，但纪律更多地表现在脚手架而不是代码中。** 保持代码库一致的工具、抽象和反馈循环越来越重要。

我们现在最困难的挑战集中在设计**环境、反馈循环和控制系统**，帮助 Agent 完成我们的目标：**大规模构建和维护复杂、可靠的软件**。

随着像 Codex 这样的 Agent 承担软件生命周期的更大部分，这些问题将更加重要。我们希望分享这些早期课程能帮助你思考在哪里投资你的精力，这样你就可以**只管构建**。
