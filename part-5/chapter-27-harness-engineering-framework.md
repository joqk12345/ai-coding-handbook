# 第27章：Harness Engineering 框架——从漂移的 Agent 到复合的系统

> LangChain 的编程 Agent 在 Terminal Bench 2.0 上的得分从 52.8% 跃升至 66.5%——从 Top 30 跃升至 Top 5——而他们没有改变任何模型参数。他们改变的是 Harness。

---

## 什么是 Agentic Engineering

我这里采用的 **agentic engineering** 定义很直接：**用 coding agents 辅助开发软件的工程实践。**

这里的 coding agent，不是只会补全几行代码的助手，而是**既能生成代码，又能执行代码**的一类 agent。Claude Code、OpenAI Codex、Gemini CLI 都属于这一范畴。它们的共同点不在于界面，而在于它们都能把“生成”与“执行”串进一个闭环。

如果再往下追问什么叫 agent，我更接受这个足够实用的定义：

> **Agents run tools in a loop to achieve a goal.**

也就是说，所谓 agent，本质上是一层软件 harness：它接收你的目标，把 system prompt、工具定义和上下文一起交给 LLM；模型请求调用工具；系统执行这些工具；再把结果送回模型；如此循环，直到任务达到某个完成条件。

对于 coding agent 来说，最关键的那类工具就是**代码执行工具**。它可以是 Bash、Python、测试运行器、浏览器自动化，或者任何能够对真实环境施加动作并返回真实结果的执行面。

### 为什么“可执行代码”是分界线

代码生成本身并不稀缺。真正让 agentic engineering 成立的，不是模型会不会写出一段看起来像样的函数，而是它能不能：

- 运行刚刚写出的代码；
- 看到真实错误；
- 基于错误继续修改；
- 反复迭代到“确实可工作”为止。

没有代码执行，LLM 的输出最多只是候选方案；有了代码执行，系统才第一次具备了**向可验证结果迭代**的能力。这是 coding agent 和传统代码补全工具之间最重要的边界。

### 这并不意味着人类没事可做

一旦 agent 能写并运行代码，真正变化的不是“工程师被替代”，而是工程师的工作重心发生转移。

写代码从来不是软件工程的全部。更核心的工作一直是：

- 定义要解决的真正问题；
- 在多个方案和约束之间做取舍；
- 给系统补足它完成任务所需的工具和上下文；
- 验证结果是否可信、是否稳健、是否真的满足业务要求；
- 在失败后更新指令、流程和 harness，让系统下次少犯同样的错。

LLM 本身不会从一次失败中自动长久学习，但 coding agent 所依赖的提示、工具和运行框架是可以被持续修正的。只要我们把教训沉淀回 system prompt、docs、skills、tooling 和 guardrails，系统就能逐步积累经验。

### 这不是 Vibe Coding

很多人会把一切“让 AI 写代码”的行为都叫作 vibe coding，但这种混用会让讨论失焦。

更准确的区分是：

- **vibe coding**：更接近“先让模型写出来再说”，强调速度、探索和原型，不严格要求过程审查和生产级可信度。
- **agentic engineering**：依然使用 coding agents，但目标是把结果带到可验证、可迭代、可交付的工程标准。

前者对探索和学习非常有价值，后者则要求你对输出负责。两者都重要，但不应该混为一谈。我们需要保留这个区别，否则就无法讨论什么叫“把 LLM 生成的代码真正带到生产可用”。

### 本章的立场

本章接下来讨论的，不是“如何优雅地把任务丢给模型”，而是如何构建一套方法，使 coding agents 在真实软件工程中持续产出可信结果。

换句话说，这一章谈的是：**当代码可以由 agent 大量生成时，人类如何通过 harness、上下文、验证与治理，把这种能力变成稳定的工程生产力。**

### 代码变便宜了，但好代码没有

agentic engineering 最难适应的一点，是要真正接受一个新事实：**把代码敲进电脑这件事，已经变得非常便宜。**

过去，代码一直是昂贵产物。一个工程师写出几百行干净、可测、可维护的代码，往往就要消耗半天到一天。围绕这个成本结构，我们形成了大量工程习惯。

在宏观层面，团队会反复做需求澄清、方案设计、排期估算和优先级管理，本质上都是为了确保昂贵的编码时间被花在最值得的地方。一个功能值不值得做，通常先看它是否配得上开发成本。

在微观层面，工程师每天也都在做大量“时间换质量”的局部权衡：

- 这个函数值不值得再重构得更优雅一点；
- 这个边界情况值不值得补一条测试；
- 这段文档值不值得现在就写；
- 这个调试界面值不值得为了未来排障先搭起来。

coding agents 让“把代码写出来”这件事的边际成本骤降，于是很多过去看似稳定的直觉开始失效。尤其是当你还能同时调度多个并行 agent，让一个人同时推进实现、重构、测试和文档时，原先那些关于“值不值得花时间”的判断会被彻底打乱。

### 几乎免费的，是代码；不是好代码

新代码的交付价格确实已经接近于零，但**好代码**远没有这么便宜。

这里的“好代码”，至少意味着：

- 它真的能工作，而不是只在 happy path 下看起来能工作；
- 我们知道它能工作，已经用测试、运行验证、审查或观测手段证明了这一点；
- 它解决的是正确的问题，而不是把需求误解成另一个更容易实现的问题；
- 它能优雅、可预期地处理错误，而不是一出异常就把未来维护者扔进黑箱；
- 它足够简单，只做真正需要做的事，并且今天的人和未来的机器都还能看懂；
- 它被测试保护着，未来改动时不会悄悄回归；
- 它有恰当层级的文档，而且文档会随着行为变化而更新；
- 它为合理的未来变化留出了空间，但又没有为了想象中的未来需求过度复杂化；
- 它满足当前项目真正重要的那些 “ilities”，例如可靠性、安全性、可测试性、可维护性、可观测性、可访问性和可扩展性。

coding agents 可以帮助完成其中大部分工作，但它们不会自动替你判断“哪一种好，才是当前项目需要的那一种好”。这个责任仍然落在驱动这些 agent 的工程师身上。

### 我们需要建立新的工程习惯

因此，agentic engineering 真正要求变化的，不只是工具链，而是个人与组织层面的决策习惯。

过去很多“不值得做”的动作，现在应该至少重新审视一遍。以前你可能会想：“算了，不值得再花一小时做这个小优化。” 现在更合理的做法常常是：**先把任务丢给 agent 跑起来，再基于结果决定值不值得继续。**

这会带来一个很实际的新工作方式：

- 对低风险、可异步验证的改进，默认更愿意尝试；
- 对测试、文档、调试工具、可观测性、辅助界面这类过去常被砍掉的“次要项”，重新评估其成本收益；
- 不再只问“值得花一个工程师半天吗”，而是问“值得花几分钟 token 和一次 review 吗”；
- 把个人直觉从“节省编码时间”转向“节省验证时间、返工时间和未来维护成本”。

今天行业里的最佳实践还远没有定型。更现实的做法，是不断反问自己：**如果我不让 agent 试一下，这个判断是不是仍然建立在旧时代“写代码很贵”的直觉上？**

### 囤积那些你知道怎么做的事

很多高产的 agentic engineering 技巧，其实只是把传统工程直觉推进到一个新阶段。其中一个非常值得保留并强化的习惯是：**囤积那些你已经知道怎么做的事。**

软件工程的一项核心能力，从来不只是“会写代码”，而是知道什么事情能做到、什么事情做不到，以及大概该怎么做。这个知识既可能很宽泛，也可能非常偏门：

- 浏览器里能不能只靠 JavaScript 做 OCR；
- iPhone 应用在不前台运行时能不能继续和蓝牙设备配对；
- Python 能不能处理 100GB 的 JSON 文件而不把它一次性读进内存。

你脑子里这类“可行性答案”越多，就越容易在别人还没想到的时候，看见技术组合解决问题的新路径。

真正有价值的，不只是“理论上可行”，而是**你见过它跑起来**。看过真实代码、跑通过原型、做过一个小 proof-of-concept，这些证据会让你的判断更扎实，也更容易被未来的 agent 复用。

### 把样例当成长期资产，而不是一次性试验

因此，很多有经验的工程师都会长期积累自己的“技术仓库”：博客笔记、TIL、demo repo、小型实验项目、只验证一个关键想法的最小原型。这些内容过去主要服务于人类自己，现在它们还多了一个新用途：**成为 coding agents 的高质量输入。**

对 agent 来说，最有用的素材之一，不是抽象解释，而是已经工作的例子。一个能跑的 HTML 工具、一段可复用的测试写法、一个最小可行的 WebAssembly demo、一个特定库的调用骨架，都比纯概念描述更容易被它拿来重组。

这意味着你过去写下来的那些“顺手存档”，今天其实变成了非常有价值的工程资产：

- 它们帮助你自己记住哪些技术路径可行；
- 它们让 agent 有现成的风格、结构和实现参考；
- 它们把“重新解释需求”变成“重组已有能力”，显著降低出错率。

### 最强的 prompting 之一：让 agent 组合多个已验证例子

其中一个非常有效的提示模式，是要求 agent **基于两个或多个已有工作样例，组合出一个新工具或新功能**。

这种方式尤其适合下面这类任务：

- 一个功能本身不复杂，但涉及两个你已经验证过的技术栈；
- 需求比较新，但底层组件你其实早就做过；
- 你不想从零解释原理，只想让系统把几块现成拼图正确拼起来。

一个典型例子是“PDF 页渲染 + OCR”的组合。单独看，这两个能力都不新：一边是把 PDF 页面转成图像，另一边是对图像做 OCR。难点不是原理，而是把两个已知可行的片段快速拼成一个真正可用的小工具。

过去你可能要手工把两个 demo 合并、调事件、改 DOM、处理交互。现在更高效的方式，往往是直接把两个工作样例交给 agent，然后明确说明你想要的新交互和最终输出，让它在已有证据之上完成组装。这样做的成功率，通常远高于让它从一段抽象需求完全白手起家。

### coding agents 把这种做法放大了

在没有 coding agent 的时候，囤积样例主要是为了你自己以后参考。现在，它的收益被进一步放大了。

如果 agent 可以访问互联网，你可以要求它：

- 抓取某个在线 HTML 工具的源码作为参考；
- 比较两个公开 demo 的实现方式，再合成一个第三种交互；
- clone 你的研究仓库，在里面找出某类技术样例，再迁移到当前项目。

如果 agent 跑在你自己的机器上，这种模式会更强，因为它很擅长搜索你本地已经积累的代码资产。你完全可以像这样给任务：

- 参考我另一个项目里 mock HTTP 测试的写法，把这套模式迁移到当前仓库；
- 去我的研究目录里找 Rust 编译到 WebAssembly 的最小案例，然后基于那个案例做一个当前项目需要的 demo；
- 参考我以前做过的 OCR 工具和 PDF 工具，组合成一个新的浏览器端处理流程。

很多时候，这样一句话就够了。agent 会自己启动搜索、阅读、抽取和重组，把真正相关的部分带回来。

### 关键不是囤积更多，而是囤积可复用的证据

这里真正重要的，不是机械地收藏更多代码片段，而是有意识地积累那些将来可能反复复用的“已验证能力单元”。

一旦某个技巧被你验证过一次，并且被保存成可读、可运行、可检索的形式，你未来就不需要再从零搞懂它第二次。你和你的 agent 都可以把它当成标准零件，在后续项目里不断复用。

从这个角度看，agentic engineering 的一个重要收益就是：**很多有用的技巧，我们只需要真正搞明白一次。** 之后只要那个技巧连同工作样例一起被妥善保存，它就能成为未来无数相似任务的输入资产。

### AI 应该帮助我们产出更好的代码

很多开发者担心，把更多编码工作外包给 AI 之后，最终只会得到质量下降但交付更快的系统，甚至让团队默认接受“反正先上线”的坏代码。

如果引入 coding agents 之后，你们交付的代码质量真的下降了，那不应该被解释为“这就是 AI 的代价”，而应该被当成一个明确的工程问题来处理：**到底是流程的哪一部分在拉低质量，然后把那一部分修掉。**

换句话说，借助 agents 交付更差的代码不是命运，而是一种选择。既然我们能选择交付更快，也完全可以选择交付更好。

### 最好的技术债治理，是尽量别欠下它

技术债的本质通常是权衡。不是不知道什么做法更正确，而是因为时间太贵，只能先走一条更快但没那么干净的路，然后寄希望于未来有机会补回来。

问题在于，很多技术债本身并不复杂，它们只是**简单但耗时**：

- 早期 API 设计漏掉了重要场景，正确修复要改很多地方，于是团队选择再叠一个稍微不同的新接口；
- 某个关键概念命名一开始就错了，但全量改名太费时间，于是只在界面上修，底层继续将错就错；
- 相似但略有差异的功能越长越多，大家都知道应该抽并重构，却一直排不上；
- 一个文件膨胀到几千行，本来应该拆模块，但总有“更紧急”的任务压着。

这些债之所以常年存在，不是因为难，而是因为在旧成本结构下很难证明“现在花时间修它们”是值得的。

### 这类重构，正适合交给 coding agents

恰好，这些“概念上简单、执行上琐碎”的任务，正是 coding agents 最适合处理的一类工作。

你可以直接启动一个 agent，让它在独立 branch 或 worktree 里慢慢跑：

- 统一改名；
- 批量调整 API 调用；
- 合并重复实现；
- 拆分超长文件；
- 顺手补文档和测试。

然后用 Pull Request 来评估结果：

- 如果结果够好，就合并；
- 如果差一点，就继续提示它调整；
- 如果明显不行，就直接丢弃。

因为这类改进的执行成本已经被压到非常低，我们终于可以对很多过去会被默许的小味道、小重复、小别扭，采取更接近 **zero tolerance** 的态度。

### AI 工具让我们可以认真比较更多方案

很多真正昂贵的技术债，其实不是出在实现阶段，而是出在更早的方案选择阶段。我们太早选了一个并不最合适的方案，或者压根没看到那个更简单、更稳妥的选项。

LLM 在这里的一个现实价值，是帮我们减少“漏掉显而易见选项”的概率。它们通常更擅长提出训练数据里那些常见、朴素、无聊但可靠的做法。很多时候，这恰恰就是最该优先考虑的方案。

更重要的是，coding agents 大幅降低了**探索型原型**的成本。

很多技术决策，最可靠的办法不是开会辩论，而是拿一个原型去证明。比如：

- Redis 到底适不适合做一个高并发站点的 activity feed；
- 某种缓存结构到底能不能扛住目标负载；
- 某个新组件在你现有代码库里到底会不会让复杂度失控。

最稳的办法通常是搭一个模拟、跑一次压测、看它到底先坏在哪里。而这种实验过去之所以做得少，往往只是因为太花时间。现在 agent 可以用一条好 prompt 快速搭出原型，多个方案甚至可以并行试。这样你不再只能选“第一个看起来能做的方案”，而是真能比较几个不同方向，挑一个最适合当前问题的。

### 拥抱复利式工程循环

agents 会遵循指令，而指令、流程和 harness 是可以持续演化的。一次任务做完之后，真正有价值的不只是那次交付本身，还有你从中学到了什么，以及这些经验是否被沉淀回系统。

Every 把这种做法叫做 **Compound Engineering**。他们的思路很简单：每个 coding project 结束后都做一次 retrospective，把这次什么有效、什么无效、以后该怎么提示和约束 agent 记录下来，作为下一次运行的输入。

这也是我认为最值得坚持的习惯之一。我们不该把 agent 当成一台每次都重新开始的随机机器，而应该把它当成一个可以不断积累工作方式的系统：

- 小的质量改进会复利；
- 过去嫌贵没做的质量投资，现在很多已经便宜到没有理由不做；
- 新功能交付和质量提升，不再必然是此消彼长的关系。

如果 harness、提示和验证闭环设计得当，coding agents 真正带来的不应该是“更快写出更多代码”，而是**更快把代码库推向更高质量的状态。**

## 行业的盲点

Gartner 预测，到 2027 年，超过 40% 的 Agentic AI 项目将被取消。七项独立研究证实，Agent 在处理复杂企业任务时的失败率高达 70-95%。

**共识观点认为模型是瓶颈。**

但当你真正测试时，模型已经成为商品。Claude、GPT-5、Gemini 2.5——它们的能力每季度都在趋同。那些正在交付生产级 Agent 的团队，胜在拥有最好的 Harness，而不是最好的模型访问权限。

**分岔路口：关键不在于你选择哪个模型，而在于你围绕模型构建什么系统。**

以下是 Harness Engineering 框架，它区分了那些漂移的 Agent 和那些能够复合价值的 Agent。

---

## 什么是 Harness

**Agent Harness 是围绕模型构建的基础设施，用于管理其运行方式——既不同于模型本身，也不同于提示词。**

OpenAI 构建了一个超过 100 万行代码的生产级应用，其中没有一行是由人类编写的。工程师们设计的是 Harness——约束、反馈循环、文档、代码检查器和生命周期管理，让模型能够可靠地编写代码。

**Harness 有三个职责：**

1. **上下文架构（Context Architecture）**——决定模型在每一步看到什么信息
2. **执行护栏（Execution Guardrails）**——强制执行模型能做什么和不能做什么
3. **记忆基础设施（Memory Infrastructure）**——确保模型从自身历史中学习

没有 Harness，模型在孤立环境中推理得很好，但在生产环境中失败。加上 Harness，你就拥有了一个能够复合价值的系统。

## 为什么巨型指令文件会失败

**典型模式：** 从一个 CLAUDE.md 或 AGENTS.md 开始，它逐渐增长成一部百科全书。每个边界情况都添加一条新规则，每次失败都添加一条新指令，最终文件达到 2000 行。模型开始忽略其中一半。

**上下文是稀缺资源。** 一个巨型指令文件挤占了任务、代码和相关文档的空间。当一切都"重要"时，就没有什么是重要的。

OpenAI 的 Codex 团队通过将他们的指令文件视为**目录**解决了这个问题。实际知识存储在一个结构化的 `docs/` 目录中。模型在正确的时间被指向正确的上下文——而不是在所有时间被淹没在所有上下文中。

**模式：**

- 保持根指令文件在 200 行以内
- 按主题在结构化目录中组织知识
- 使用**动态上下文注入**——为每个任务加载相关文档，而不是每个会话加载所有内容
- **积极修剪。** 如果一条规则在 30 天内没有阻止失败，就删除它

## 四大支柱

Harness Engineering 建立在四大支柱之上。传统设置构建一两个。生产环境需要全部四个。

### 支柱1：上下文架构（Context Architecture）

**分层、渐进式披露。** 模型首先看到项目级上下文，然后是模块级，然后是文件级。它从不一次性看到所有内容。

**模式：**

| 层级 | 内容 | 加载时机 |
|------|------|----------|
| Layer 1 | 项目架构、约定、不变量 | 始终加载 |
| Layer 2 | 模块特定文档、模式、约束 | 按任务加载 |
| Layer 3 | 文件历史、最近变更、相关测试结果 | 按文件加载 |

**上下文预算：** 跟踪每层的 token 使用量，当任何层超过窗口的 40% 时发出警报。

### 支柱2：Agent 专业化（Agent Specialization）

**单一通用 Agent 无法扩展。** 生产系统使用专业化 Agent——职责狭窄，工具范围限定。

**模式：**

- 每个领域一个 Agent：代码生成、测试、审查、部署
- 每个 Agent 只获得它需要的工具——没有通用工具访问权限
- Agent 通过**结构化交接**通信，而不是共享上下文窗口
- 按 Agent 角色限定系统提示词。编写测试的 Agent 不需要部署指令

### 支柱3：持久记忆（Persistent Memory）

**会话历史在窗口关闭时消失。** 那不是记忆。真正的记忆存在于文件系统中。

**模式：**

| 文件 | 内容 |
|------|------|
| `decisions.md` | 架构决策及其理由和日期 |
| `failure-catalog.md` | 索引化的失败模式及其解决方案模式 |
| `session-state.md` | 上次会话改变了什么、哪些测试失败了、Agent 学到了什么 |

- Agent 在会话开始时读取记忆
- Agent 在会话结束时将结果写入记忆
- 记忆是**仅追加**的，用于审计追踪。通过单独的审查流程修剪

### 支柱4：结构化执行（Structured Execution）

**模型不应该从提示词直接跳到代码。** 结构化执行强制执行工作流：研究 → 计划 → 执行 → 验证。

**模式：**

| 步骤 | 活动 |
|------|------|
| Step 1 | Agent 读取相关上下文和记忆 |
| Step 2 | Agent 生成计划（要改哪些文件、要写哪些测试、存在哪些风险） |
| Step 3 | 在关键路径上，执行前审查计划 |
| Step 4 | Agent 在护栏下执行（每会话最大文件数、受保护文件列表、部署前模拟） |
| Step 5 | Agent 运行验证（测试、代码检查、安全扫描） |
| Step 6 | Agent 将会话结果写入记忆 |

## 护栏层级（The Guardrail Hierarchy）

护栏作为 Agent 无法覆盖的硬性策略执行——要像对待硬性政策执行一样对待它们。

```
guardrail-stack/
├── hard-limits/           # 硬性限制
│   ├── cost-ceiling.md    # 每会话最大花费
│   ├── duration-limit.md  # 自动暂停前的最大运行时间
│   ├── file-budget.md     # 每会话最大变更文件数
│   └── tool-allowlist.md  # 只有这些工具可用
├── safety-nets/           # 安全网
│   ├── simulation-gate.md # 每次部署前模拟
│   ├── test-gate.md       # 提交前测试必须通过
│   ├── review-gate.md     # 关键路径人工审查
│   └── rollback-plan.md   # 失败时自动回滚
├── golden-paths/            # 黄金路径
│   ├── code-patterns.md     # 每门语言的批准模式
│   ├── architecture-rules.md # 系统边界约束
│   └── naming-conventions.md # 一致的命名执行
└── audit/                   # 审计
    ├── action-log.md          # 每个 Agent 动作带时间戳
    ├── cost-tracker.md        # 累积花费追踪
    └── decision-trail.md     # Agent 为何选择每个动作
```

**层级很重要。** 硬性限制不能被 Agent 放宽。安全网在失败到达生产环境前捕获它们。黄金路径引导 Agent 走向正确的模式。审计确保你可以重建发生了什么。

## 复利效应

没有 Harness 的 Agent 每次会话都从零点开始——重新发现模式、重复错误、不带任何机构知识。

有 Harness 的 Agent **复合价值**。

每次会话都添加到失败目录，每个决策都记录理由，每个交互模式都被索引以供重用。

OpenAI 的 Codex 团队报告称，他们的 Harness 驱动 Agent 随时间变得更加可靠——不是因为模型改进了，而是因为 Harness 积累了让模型工作更轻松的上下文。

**这才是真正的护城河。** 模型是可互换的。Harness 不是。六个月积累的上下文、失败模式和架构决策无法通过切换到新模型或新框架来复制。

## 生产环境检查清单

### 上下文架构
- [ ] 根指令文件在 200 行以内
- [ ] 知识按主题组织在结构化目录中
- [ ] 动态上下文注入按任务加载相关文档
- [ ] 每层上下文预算带警报追踪
- [ ] 每月修剪过时规则

### Agent 专业化
- [ ] 代码生成、测试、审查的独立 Agent
- [ ] 每个 Agent 有作用域工具访问（无通用工具）
- [ ] Agent 间结构化交接文档化
- [ ] 按 Agent 角色限定系统提示词

### 持久记忆
- [ ] 决策、失败、会话状态存储在文件系统中
- [ ] Agent 会话开始时读取记忆
- [ ] Agent 会话结束时写入记忆
- [ ] 记忆仅追加，单独修剪流程

### 结构化执行
- [ ] 研究 → 计划 → 执行 → 验证 工作流强制执行
- [ ] 关键路径执行前计划审查
- [ ] 每会话最大文件预算强制执行
- [ ] 受保护文件列表防止未授权修改
- [ ] 每次执行步骤后验证运行

### 护栏
- [ ] 每会话成本上限强制执行
- [ ] 持续时间限制带自动暂停
- [ ] 每次部署动作前模拟
- [ ] 任何提交前测试必须通过
- [ ] 每个 Agent 动作完整审计日志

## 对构建者的意义

**模型层正在快速商品化。** GPT-5、Claude Opus、Gemini 2.5——能力差距每季度都在缩小。卡内基梅隆大学发现 Agent 仅能完成 24% 的标准办公任务，而失败源于基础设施，而非智能。

**获胜的团队正在构建 Harness，而非购买模型。** LangChain 证明了这一点：相同模型，更好的 Harness，Top 5。

**随着模型改进，Harness Engineering 变得更加重要**——因为更好的模型解锁更多自主性，而更多自主性需要更好的护栏。

**Harness 是产品。模型是引擎。没人仅仅为了引擎而买车。**
