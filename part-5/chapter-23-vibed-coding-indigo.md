## 第一部分：杰文斯悖论与软件工程

### 什么是杰文斯悖论？

> *当一种资源变得更便宜、更容易获取时，人们往往会更多地使用它，而不是减少使用。*

的例子：
- 电价便宜了，家用电器的使用量反而增加
- 汽车普及了，交通拥堵反而更严重
- AI token 便宜了，消耗量反而爆炸式增长

### 软件工程领域的杰文斯悖论

有人担心 AI 会导致软件工程师失业，但从数据上看：

**Indeed 招聘网站显示：软件工程师的招聘岗位不降反升。**

这是为什么？

**中小企业的胆子变大了。**

以前只有大厂才能组建豪华技术团队，现在：
- 100人公司 → 雇 2 个工程师 + AI Agent 就能完成内部系统
- 小公司老板 → 自己就能上手做产品
- 以前需要外包 SaaS 的需求 → 现在可以自建

> *以前雇一个软件工程师需要 $6,000+/月，现在同样的预算可以让 AI 24小时不间断工作。*

## 第二部分：AI Agent 的本质

### 什么是真正的 Agent？

在这次直播中，主播提出了一个重要观点：

> *Claude Code 是第一个真正意义上的 Agent。*

在他看来，之前的各种"智能助手"都不能算真正的 Agent，因为：
1. 权限受限（只能在一个受限的 chat 界面工作）
2. 无法自主调用工具
3. 无法完成"使命必达"的任务

而 Claude Code 的突破在于：
- **本地化运行** - 在你的电脑上运行，拥有本地文件系统的全部权限
- **Tool Calling** - 可以调用各种工具，帮你完成实际工作
- **Background Execution** - 可以在后台持续运行

### AI Harness：给 Agent 配装备

一个新概念被提出：**AI Harness（AI 马具）**

所谓 Harness，就是你在完成一项具体任务时，给 Agent 配备的：
- 工具集（Tools）
- 场景设置（Context）
- 工作流程（Workflow）

> *这类似于古代战场上的盔甲和武器——好的 harness 让 Agent 如虎添翼。*

## 第三部分：现场 Demo 回顾

### Demo 1：用 Claude Code 搭建 Telegram 机器人

**需求**：让本地运行的 Claude Code 可以通过 Telegram 对话

**过程**：
1. 用自然语言描述需求
2. Claude Code 自主启动 3 个 sub-agent 并行工作
3. 一个 agent 调研 Claude Code 如何连接外部服务
4. 一个 agent 调研 Telegram Bot API
5. 一个 agent 编写服务代码

**结果**：约 20 分钟后，一个可工作的 Telegram 机器人就搭建完成了。

**关键洞察**：
- 不要期望一句话就完成复杂任务
- 需要和 Agent 多轮对话，明确需求
- 给 Agent 一个清晰的 context，它能做得更好

### Demo 2：用 Pencil 做 Web Design

**工具**：Pencil（一个给 Agent 用的设计画布工具）

**过程**：
1. 先建立一套 Design System（VI、品牌规范）
2. 告诉 Agent 需要什么类型的页面
3. Agent 自主在画布上设计，并生成代码

**结果**：
- 3 个 Agent 同时工作
- 20 分钟完成 3 个页面的设计（ Landing Page、Article Page、Index Page）
- 还生成了 Mobile 版本和 Light/Dark 版本

**关键洞察**：
- AI 不只是写代码，还能做设计
- 给 Agent 足够的约束（Design System），它能做得更好
- Agent 之间的协作是自然发生的

## 第四部分：未来软件形态

### 软件的四次进化

1. **软盘时代**：软件装在 5.25 寸软盘里
2. **下载时代**：通过互联网下载软件
3. **SaaS 时代**：云端访问软件，按需使用
4. **AI Agent 时代**：AI Model + Context + Agent

### 未来的软件会怎样？

> *所有能用电脑完成的工作都是软件。AI 能写所有软件 = AI 能做任何事情。*

**等式**：
```
Everything is software + AI can write all software = AI can do everything = AGI
```

未来的软件形态：
- **AI Model** - 大脑
- **Context** - 记忆和知识
- **Agent** - 执行者
- **Human** - Connector（连接者，定义需求的人）

### 人的新角色：从生产者到连接者

> *以前软件工程师是生产者，以后更像老板——分配任务，验收结果。*

这可能是人类历史上首次大规模脱离生产环节。我们需要重新定义自己的价值。

## 第五部分：实践经验总结

### 1.  不要省模型的钱

> *在这个阶段，不要用便宜模型。用最好的模型，感受智能的边界在哪里。*

- 4.5 → 直接上 4.6 或 5.4
- 便宜模型看似省钱，实际浪费更多 token 在扯皮上

### 2.  给 Agent 足够的 Context

- 写清楚你要什么
- 给足够的背景信息
- 允许它自己探索

### 3.  不要设限太死

> *一定不要预设路径，让 Agent 发挥它的智能。*

- 你只需要告诉它"在这个环境里完成这个任务"
- 不要告诉它"第一步干什么，第二步干什么"

### 4.  复用 Skills 和 Design System

- 把常用流程封装成 Skills
- 把设计规范封装成 Design System
- 这些"约束"不是限制，而是赋能

### 5.  用语音输入

面对 AI 下指令时，文字输入太慢。用语音输入 + AI 转文字，效率提升 10 倍。

## 第六部分：关于工具选择的争论

### Claude Code vs Cursor vs Codex

直播中提到：

> *如果你已经在用 Claude Code，没有必要换到 Codex。如果你用 Codex，也没有必要换。*

每个工具都有自己的生态，选择一个然后深耕，胜过换来换去。

### 什么情况下换工具？

- 当你发现某个工具的边界明显不够用时
- 当你需要它的生态（比如特定的 MCP）

### 关于"AI 味"

有些人担心 AI 生成的内容有"AI 味"——千篇一律，没有灵魂。

解决之道：
- 给 Agent 足够的风格指引
- 用你过去的作品训练 Agent 理解你的风格
- 不断挑战它，让它进化
