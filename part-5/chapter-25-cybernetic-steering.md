# 第25章：从“拧阀门”到“掌舵”——Harness Engineering 的控制论转向

> 读完 OpenAI 关于 harness engineering 的实践后，一个熟悉的模式浮现出来：这不是第一次工程角色迁移，而是历史上第三次“闭环上移”。

## 1. 一个模式，三次出现

这条线索可以被清晰地串起来：

1. **詹姆斯·瓦特的离心调速器（18世纪）**
   - 调速器出现之前，人要站在蒸汽机旁手动调阀；
   - 调速器出现之后，飞球机构自动感知转速并调节阀门；
   - 人并未消失，而是从“拧阀门”转向“设计调速器”。

2. **Kubernetes 控制器（现代软件基础设施）**
   - 工程师声明 desired state（例如副本数、镜像、资源限制）；
   - 控制器持续观察 actual state；
   - 偏离就自动 reconcile（重启、扩缩容、回滚）。
   - 角色从“手工救火”转为“定义系统应当自动收敛到的规格”。

3. **Harness Engineering（Agentic Engineering）**
   - 工程师不再把主要精力放在逐行写代码；
   - 而是设计环境、搭建反馈回路、编码架构约束；
   - 再由 agent 执行实现。

这正是 Norbert Wiener 在 1948 年提出的控制论（cybernetics）核心：
**你不再持续直接操作执行器，而是成为“舵手”（steersman）。**

## 2. 为什么代码库长期是“闭环上移”的最后堡垒？

代码工程一直有反馈回路，但大多停留在较低层：

- 编译器：校验语法与类型；
- 测试：校验行为是否符合预期；
- Linter：校验风格和静态规则。

这些回路都非常重要，但它们只能覆盖“可机械判定”的属性。真正高层且关键的问题长期缺乏自动闭环能力：

- 改动是否符合架构演进方向？
- 这个抽象在规模扩大后会不会失控？
- 当前方案是否是系统层面的正确解？

过去这类问题只能由人类同时承担“传感器 + 执行器”双重角色：
既要判断质量，也要亲手改代码。

LLM 的变化在于：

- **感知能力上移**：能读懂系统层意图与结构问题；
- **执行能力上移**：能在同一层级重构模块、重塑接口、重写测试策略。

于是，高价值决策层首次出现“可闭环”的可能。

## 3. 闭环能跑起来，只是起点

让 agent 跑通基本闭环只是门槛：

- 可执行测试；
- 可解析 CI 输出；
- 能定位问题的错误信息。

真正难点是**校准（calibration）**：
把你团队对“什么是好代码”的判断，转化成 agent 可读取、可执行、可验证的系统知识。

常见误判是：
> “Agent 总做错，它不理解我们的代码库。”

更常见的真实原因是：
> 关键知识只在人的脑子里，没有被外化。

如果架构品味、依赖方向、禁忌模式、评审标准都没有机器可读表达，agent 不会“靠次数自发对齐”，它只会稳定地重复同类错误。

## 4. Harness Engineering 的核心工作：把判断变成机器可读

可落地的外化方式包括：

1. **架构文档**：明确分层边界、依赖方向、模块职责；
2. **定制 lint / policy**：不仅报错，还给 remediation（修复建议）；
3. **Golden Principles**：把团队审美与取舍准则写成可引用规则；
4. **高频反馈基础设施**：让 agent 的每次尝试都能快速得到高质量信号。

OpenAI 的实践也印证了这一点：如果不把标准编码进 harness，团队会持续花时间清理“AI slop”；一旦标准进入闭环，质量才会随规模提升而稳定。

## 5. 为什么这是“唯一前进路径”？

文档化、自动化测试、架构约束、快速反馈，这些从来都不是新主张。过去团队可以拖延它们，是因为惩罚慢且分散：

- 质量缓慢下滑；
- onboarding 成本上升；
- 技术债悄悄复利。

在 agentic engineering 时代，惩罚变成高并发、持续性的：

- 缺文档：agent 在每一次改动中持续偏离约定；
- 缺测试：反馈环路无法闭合；
- 缺约束：架构漂移以机器速度扩散。

更关键的“反身性陷阱”是：
**如果你没有定义“干净”的标准，就无法指望 agent 把系统清理干净。**

## 6. 人类角色的再定义：从“比机器更会写”到“比机器更会判”

从 generation-verification asymmetry 的角度看（“生成正确解通常比验证正确性更难”），人类优势将更集中在：

- 明确“正确”的可验证定义；
- 快速识别偏离与伪正确；
- 判断当前方向是否值得继续投资。

这并不是人从工程中退场，而是职责上移：
从直接实现细节，转向设计可持续收敛的控制系统。

正如调速器时代的工人并没有回去手拧阀门，不是因为不会，而是因为那不再是最高价值工作。

## 7. 实践清单（可直接执行）

如果你准备把团队迁移到 harness engineering，可从以下顺序开始：

1. 先补“最低闭环”：可运行测试 + 可解析 CI + 结构化错误输出；
2. 再补“校准资产”：架构规则、编码准则、评审标准文档化；
3. 将规则代码化：lint、检查器、模板、脚手架；
4. 以小范围仓库试点，度量返工率、缺陷率、交付周期；
5. 每周回收高频错误，把新规则持续并入 harness。

把“经验”变成“系统”，把“救火”变成“掌舵”，这是第25章想强调的核心。
